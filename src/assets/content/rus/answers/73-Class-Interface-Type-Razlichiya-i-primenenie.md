
<p class="info info--blue">
  Главное правило: type и interface существуют только до сборки (Compile-time) и стираются. class — это реальная сущность,
  которая остается в итоговом JavaScript-коде (Runtime).
</p>

<h3>1. Type (Псевдоним)</h3>
<p>
  Для чего: Любые структуры (примитивы, объекты, массивы, <code>Unions |</code>, <code>Intersections &</code>,
  <code>Tuples []</code>).
</p>

<p><strong>Особенность:</strong> Статичен. После объявления нельзя расширить (нет declaration merging).</p>

<p>
  <strong>Применение:</strong> Типизация стейта, сложных составных типов, объединений (например,
  <code>type Status = 'ok' | 'error'</code>).
</p>

<h3>2. Interface (Контракт)</h3>
<p>Для чего: Описание формы только объектов.</p>

<p>
  <strong>Особенность:</strong> Открыт для расширения. Поддерживает Declaration Merging (два интерфейса с
  одинаковым именем в одном файле автоматически сливаются в один). Наследуется через extends.
</p>

<p><strong>Применение:</strong> ООП-контракты (через implements), типизация публичного API и библиотек.</p>

<p class="info info--orange">
  <strong>Best practice:</strong> использовать interface для описания форм объектов, особенно если предполагается расширение в
  будущем.
  <br>
  Type — для примитивов, объединений, кортежей и сложных типов (Union, Utility Types, Tuple),
  которые не являются просто формой объекта.
</p>

<h3>3. Class (Реализация)</h3>
<p>Для чего: Шаблон объекта, содержащий реальный рабочий код.</p>

<p>
  <strong>Особенность:</strong> Работает одновременно как тип для переменной и как значение для создания экземпляра
  (через new). Поддерживает модификаторы доступа (private/protected/public).
</p>

<p>
  <strong>Применение:</strong> Сущности с внутренней логикой и состоянием, токены для внедрения зависимостей
  (Dependency Injection).
</p>
