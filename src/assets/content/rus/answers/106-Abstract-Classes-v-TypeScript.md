<h3>Что такое абстрактный класс (Abstract Class)?</h3>
<p><span class="accent">Абстрактный класс</span> — это базовый класс, который нельзя инстанцировать напрямую (создать через <code>new</code>). Он служит «чертежом» для других классов.</p>

<p>Его главная фишка в том, что он может содержать как объявления методов (абстрактные методы, которые обязаны реализовать наследники), так и готовую реализацию (обычные методы, которые наследники получают из коробки).</p>

<p class="info info--orange">При проектировании архитектуры выбор между абстрактным классом и интерфейсом - это классическая дилемма. В TypeScript оба инструмента предоставляют мощные возможности для определения контрактов и обеспечения типовой безопасности, но они служат разным целям и имеют свои уникальные особенности. В этом ответе мы рассмотрим ключевые особенности абстрактных классов в TypeScript.</p>

<h3>Когда использовать Абстрактные классы вместо Интерфейсов?</h3>
<h4>Используй абстрактные классы, когда:</h4>

<p>1. <strong>Есть общая базовая логика.</strong> У тебя есть несколько родственных классов, которые делают часть работы одинаково. Эту общую работу (реализацию) ты выносишь в абстрактный класс, чтобы избежать дублирования кода (принцип DRY).</p>

<p>2. <strong>Нужно состояние.</strong> Базовой сущности нужно хранить какие-то внутренние переменные или флаги, которые будут использоваться во всех наследниках.</p>

<p>3. <strong>Нужны модификаторы доступа.</strong> Тебе нужно скрыть часть внутренней логики базового класса от внешнего мира, используя protected, оставив ее доступной только для классов-наследников.</p>

<p>4. <strong>Dependency Injection (Особенно актуально для фреймворков).</strong> Поскольку интерфейсы исчезают в JavaScript рантайме, их нельзя использовать как токены для внедрения зависимостей. Абстрактные классы остаются в JS, поэтому отлично работают как DI-токены (частый трюк в серьезной архитектуре).</p>

<h4>Используй интерфейсы, когда:</h4>

<p>1. Нужно просто описать «форму» объекта (например, типизировать JSON-ответ от API).</p>

<p>2. Нужно задать строгий контракт поведения для классов, которые вообще никак не связаны друг с другом логически (например, интерфейс Logger, который может быть реализован и сервисом, и компонентом).</p>

<p>3. Нужно реализовать множественное наследование поведений.</p>

<h3>Пример из жизни: Паттерн "Шаблонный метод"</h3>
<p>Абстрактные классы идеально подходят для паттерна <strong>Template Method</strong>, где базовый алгоритм задан, а детали делегированы наследникам.</p>

<code class="code">
  abstract class DataFetcher {
    // Общее состояние
    protected isFetching = false;

    // Готовая реализация (общая для всех)
    public async fetchAndProcess(): Promise<void> {
      this.isFetching = true;
      try {
        const data = await this.getData(); // Вызов абстрактного метода
        this.processData(data);            // Вызов абстрактного метода
      } finally {
        this.isFetching = false;
      }
    }

    // Наследники ОБЯЗАНЫ реализовать эти детали
    protected abstract getData(): Promise<any>;
    protected abstract processData(data: any): void;
  }

  // Конкретная реализация
  class UserFetcher extends DataFetcher {
    protected async getData(): Promise<any> {
      return fetch('/api/users').then(res => res.json());
    }

    protected processData(data: any): void {
      console.log('Обработка пользователей:', data);
    }
  }

  const fetcher = new UserFetcher();
  fetcher.fetchAndProcess(); // Используем общий алгоритм с уникальными деталями
</code>
