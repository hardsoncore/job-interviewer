<p>
  <span class="accent">Intersection Observer API</span> — это современный веб-API, который позволяет отслеживать пересечение элемента с другим элементом или с областью видимости (viewport). Он предоставляет способ асинхронного наблюдения за изменениями в пересечении элемента с его родительским элементом или с областью видимости, что особенно полезно для оптимизации производительности и улучшения пользовательского опыта.
</p>

<p class="info info--blue">
  Статус «кроссбраузерного стандарта» API получил в марте 2019 года. Сегодня это абсолютно базовый функционал, который смело можно использовать в любых современных браузерах.
</p>

<h3>Основные юзкейсы Intersection Observer API:</h3>

<ul>
  <li>
    <strong>Ленивая загрузка (Lazy Loading):</strong> Изображений, iframe'ов или тяжелых компонентов, когда они попадают в зону видимости.
  </li>
  <li>
    <strong>Бесконечный скролл (Infinite Scrolling):</strong> Подгрузка новых данных при достижении конца списка.
  </li>
  <li>
    <strong>Анимации при скролле:</strong> Запуск CSS-анимаций ровно в тот момент, когда элемент появляется на экране.
  </li>
  <li>
    <strong>Аналитика видимости:</strong> Отслеживание того, увидел ли пользователь рекламный баннер или определенный блок текста.
  </li>
</ul>

<h3>Архитектура и основные понятия</h3>
<p>
  Чтобы начать работу, нужно создать экземпляр <code>IntersectionObserver</code>, передав в него callback-функцию и объект с настройками (options).
</p>

<h4>1. Настройки (Options)</h4>

<ul>
  <li>
    <strong>root:</strong> Элемент, который выступает в роли области просмотра (viewport). По умолчанию — окно браузера (<code>null</code>). Если указываешь конкретный элемент, он должен быть предком отслеживаемого элемента.
  </li>
  <li>
    <strong>rootMargin:</strong> Отступы вокруг root элемента. Работает как CSS margin (например, <code>"10px 20px 30px 40px"</code>, <code>"50px"</code> или <code>"10%"</code>). Позволяет расширить или сузить область срабатывания. Полезно, если нужно начать загрузку картинки за <code>50px</code> до того, как она фактически появится на экране.
  </li>
  <li>
    <strong>threshold:</strong> Чувствительность к пересечению. Число или массив чисел от <code>0.0</code> до <code>1.0</code>. Указывает, какой процент элемента должен быть виден, чтобы сработал callback.
    <br>
    Например, <code>[0, 0.5, 1]</code> даст возможность вызывать callback при пересечении элемента на 0%, 50% и 100%.
  </li>
</ul>

<h4>2. Callback-функция</h4>
<p>
  Функция, которая вызывается при пересечении. Принимает два аргумента:
</p>

<ul>
  <li><strong>entries:</strong> Массив объектов <code>IntersectionObserverEntry</code> (данные о каждом отслеживаемом элементе).</li>
  <li><strong>observer:</strong> Ссылка на сам экземпляр наблюдателя.</li>
</ul>

<p>
  Главные свойства объекта <code>entry</code> (из массива <code>entries</code>):
</p>

<ul>
  <li><strong>isIntersecting (boolean):</strong> пересекается ли target элемент с <code>root</code> прямо сейчас с учетом <code>threshold</code>.</li>
  <li><strong>intersectionRatio (number):</strong> текущий процент видимости (от 0.0 до 1.0).</li>
  <li><strong>target (Element):</strong> сам DOM-элемент, за которым ведется наблюдение.</li>
  <li><strong>boundingClientRect (DOMRectReadOnly):</strong> размеры и позиция target.</li>
  <li><strong>intersectionRect (DOMRectReadOnly):</strong> размеры и позиция видимой части target.</li>
  <li><strong>rootBounds (DOMRectReadOnly):</strong> размеры и позиция root.</li>
  <li><strong>time (DOMHighResTimeStamp):</strong> время, когда произошло пересечение (полезно для аналитики).</li>
</ul>

<h3>Пример использования</h3>

<code class="code">
  // 1. Задаем настройки
  const options = {
    root: null, // viewport браузера
    rootMargin: '0px',
    threshold: 0.5 // Сработает, когда видно 50% элемента
  };

  // 2. Создаем callback
  const callback = (entries, observer) => {
    entries.forEach(entry => {
      if (entry.isIntersecting) {
        console.log('Элемент в зоне видимости!', entry.target);

        // Выполняем логику (например, подгружаем картинку)
        // entry.target.src = entry.target.dataset.src;

        // ОЧЕНЬ ВАЖНО: прекращаем наблюдение, если действие нужно выполнить один раз
        // Это предотвращает утечки памяти в SPA (Vue/Angular)
        observer.unobserve(entry.target);
      }
    });
  };

  // 3. Инициализируем Observer
  const observer = new IntersectionObserver(callback, options);

  // 4. Находим элемент и начинаем наблюдение
  const targetElement = document.querySelector('.my-element');
  observer.observe(targetElement);

  // При уничтожении компонента (например, в ngOnDestroy или beforeDestroy)
  // не забывай вызывать observer.disconnect() для полной очистки.
</code>

<h3>Механика работы Intersection Observer API</h3>

<h4>Когда вызывается колбек?</h4>
<p>
  При создании <code>IntersectionObserver</code>, мы будем каждый раз вызывать колбек, который передали при создании, когда выполняется условие, описанное в options (при пересечении порога <code>threshold</code>).
</p>

<p>
  Колбек будет получать массив объектов <code>IntersectionObserverEntry</code>, каждый из которых содержит информацию о том, пересекается ли элемент с <code>root</code>, насколько он виден (<code>intersectionRatio</code>) и сам элемент (<code>target</code>).
</p>

<p>
  Внутри колбека мы можем проверить, пересекается ли элемент с помощью свойства <code>isIntersecting</code>. Если оно <code>true</code>, значит элемент виден в зоне видимости (или внутри <code>root</code> элемента) на определенный процент, который мы указали в <code>threshold</code>.
</p>

<p>
   Работает в обе стороны — и когда элемент появляется, и когда исчезает из зоны видимости.
</p>

<p>
  Работает бесконечно. Если элемент появляется и исчезает, колбек будет вызываться каждый раз, когда выполняются условия пересечения. Поэтому важно контролировать, когда нужно прекратить наблюдение (например, после первой загрузки картинки), чтобы не создавать лишнюю нагрузку на браузер.
</p>

<h4>Массив entries. Что внутри?</h4>

<p>
  Это список только тех элементов, за которыми мы явно приказали следить (через <code>observer.observe()</code>), и состояние которых только что изменилось (они пересекли <code>threshold</code>).
</p>

<p>
  Браузер собирает их в массив для оптимизации: если при быстром скролле порог пересекли сразу 10 элементов, браузер отдаст их пачкой за один вызов коллбэка, чтобы не блокировать поток.
</p>

<p>
  Внутри массива лежат не чистые HTML-теги, а объекты-снимки состояния. Сам DOM-узел хранится в свойстве <code>entry.target</code>, а статус видимости — в <code>entry.isIntersecting</code>.
</p>

<h3>Best Practices</h3>

<ul>
  <li>Всегда отключай наблюдение (<code>unobserve</code> или <code>disconnect</code>) для элементов, которые больше не нужны, чтобы избежать утечек памяти.</li>
  <li>Будь осторожен с большим количеством наблюдаемых элементов — это может повлиять на производительность. Не создавай новый <code>IntersectionObserver()</code> для каждой карточки в списке из 1000 элементов. Создай один инстанс и вызови <code>observer.observe(target)</code> в цикле для всех карточек.</li>
  <li>Хоть вычисление пересечений и идет асинхронно, сам callback выполняется в главном потоке. Не делай внутри него тяжелых синхронных вычислений. Если нужна сложная логика, используй requestIdleCallback или setTimeout.</li>
</ul>
