<p class="info info--blue">
  Это операторы-преобразователи, которые делают две вещи:
  <br>
  <br>
  <strong>Проекция (Mapping):</strong> На КАЖДОЕ значение из «внешнего» потока они создают новый «внутренний» поток (inner Observable).
  <br>
  <strong>Управление подпиской (Flattening):</strong> Они автоматически подписываются на эти внутренние потоки и выводят их значения в общий поток.
  <br>
  <br>
  Разница между ними заключается только в стратегии конкурентности: что делать, если пришло новое значение, пока старый внутренний поток ещё не завершился?
</p>

<h3>mergeMap</h3>
<p>
  <strong>mergeMap</strong> (или flatMap) подписывается на каждый внутренний Observable и объединяет их результаты в один поток. Это означает, что все внутренние Observable выполняются параллельно, и их результаты смешиваются в одном потоке.
</p>

<p>
  <strong>Когда использовать:</strong> независимые операции, порядок не важен — скачивание файлов, сохранение несвязанных записей, websocket-фреймы.
</p>
<p>
  <strong>Потенциальные проблемы:</strong> при высокочастотном outer-потоке можно создать бесконечное число открытых HTTP-запросов. Используй <code>mergeMap(fn, concurrency)</code> чтобы ограничить параллелизм.
</p>

<code class="code">
  import { fromEvent, interval } from 'rxjs';
  import { mergeMap, take } from 'rxjs/operators';

  const clicks = fromEvent(document, 'click');
  const result = clicks.pipe(
    mergeMap((ev) => interval(1000).pipe(take(3)))
  );
  // При каждом клике запускается новый interval, который
  // генерирует последовательность чисел (0, 1, 2 т.д.) каждую секунду
  // все выполняются параллельно

  result.subscribe(x => console.log(x));
  // Вывод: при каждом клике будет выводиться 0, 1, 2 с интервалом в 1 секунду,
  // и эти потоки будут смешиваться
</code>

<h3>concatMap</h3>
<p>
  <strong>concatMap</strong> подписывается на каждый внутренний Observable, но выполняет их последовательно. Он ждет, пока текущий внутренний Observable завершится, прежде чем подписаться на следующий. Это гарантирует порядок выполнения, но может быть медленнее, если внутренние Observable выполняются долго.
</p>

<p>
  <strong>Когда использовать:</strong> анимации (следующая — после окончания текущей), последовательные операции с зависимостями, мигрирование данных, воспроизведение действий.
</p>
<p>
  <strong>Потенциальные проблемы:</strong> если inner-Observable никогда не завершается (<code>complete</code> не приходит), очередь заблокирована навсегда.
</p>

<code class="code">
  import { fromEvent, interval } from 'rxjs';
  import { concatMap, take } from 'rxjs/operators';

  const clicks = fromEvent(document, 'click');
  const result = clicks.pipe(
    concatMap((ev) => interval(1000).pipe(take(3)))
  );
  // При каждом клике запускается новый interval, который
  // генерирует последовательность чисел (0, 1, 2 т.д.) каждую секунду

  // Но в отличие от mergeMap, эти потоки выполняются последовательно:
  // следующий начинается только после завершения предыдущего

  result.subscribe(x => console.log(x));
  // Вывод: при каждом клике будет выводиться 0, 1, 2 с интервалом
  // в 1 секунду, но следующий набор чисел начнется только после завершения предыдущего
</code>

<h3>switchMap</h3>
<p>
  <strong>switchMap</strong> подписывается на каждый внутренний Observable, но отменяет предыдущий, если новый Observable приходит до завершения предыдущего. Это полезно для сценариев, где важен только последний результат, например, при поисковых запросах.
</p>

<p>
  <strong>Когда использовать:</strong> поисковые запросы по вводу, автодополнение, навигация по роутам — любой сценарий где старый результат теряет актуальность. Именно поэтому это оператор по умолчанию в большинстве Angular HttpClient-паттернов.
</p>
<p>
  <strong>Потенциальные проблемы:</strong> не используй с мутирующими операциями (POST, DELETE). Если запрос на сервер был отменён в RxJS, сетевой запрос браузера всё равно улетел — отмена происходит только на уровне подписки на ответ.
</p>

<code class="code">
  import { fromEvent, interval } from 'rxjs';
  import { switchMap, take } from 'rxjs/operators';

  const clicks = fromEvent(document, 'click');
  const result = clicks.pipe(
    switchMap((ev) => interval(1000).pipe(take(3)))
  );
  // При каждом клике запускается новый interval, который
  // генерирует последовательность чисел (0, 1, 2 т.д.) каждую секунду

  // Но если клик происходит до завершения текущего interval,
  // предыдущий поток будет отменён и начнётся новый

  result.subscribe(x => console.log(x));
  // Вывод: при каждом клике будет выводиться 0, 1, 2 с интервалом в
  // 1 секунду, но если кликнуть снова до завершения текущего потока,
  // он будет отменён и начнётся новый
</code>


<h3>exhaustMap</h3>
<p>
  <strong>exhaustMap</strong> подписывается на первый внутренний Observable и игнорирует все последующие, пока текущий не завершится. Это полезно для предотвращения одновременного выполнения нескольких внутренних Observable, например, при обработке кликов.
  <br>
  Пока активен текущий inner-Observable, все новые значения из outer просто игнорируются.
</p>

<p>
  <strong>Когда использовать:</strong> submit-кнопки (double-submit protection), логин, оплата — когда повторный клик до завершения первого должен быть проигнорирован.
</p>
<p>
  <strong>Потенциальные проблемы:</strong> из-за того что нажатия теряются, пользователь может не получить обратной связи. Лучше комбинировать с блокировкой UI на время запроса.
</p>

<code class="code">
  import { fromEvent, interval } from 'rxjs';
  import { exhaustMap, take } from 'rxjs/operators';

  const clicks = fromEvent(document, 'click');
  const result = clicks.pipe(
    exhaustMap((ev) => interval(1000).pipe(take(3)))
  );
  // При каждом клике запускается новый interval, который
  // генерирует последовательность чисел (0, 1, 2 т.д.) каждую секунду

  // Но если клик происходит до завершения текущего interval,
  // новый поток будет проигнорирован

  result.subscribe(x => console.log(x));
  // Вывод: при каждом клике будет выводиться 0, 1, 2 с интервалом в
  // 1 секунду, но повторные клики до завершения текущего потока будут игнорироваться
</code>
