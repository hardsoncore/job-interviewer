<p>
  Современные JS-движки (будем рассматривать на примере самого популярного — V8 от Google) используют модель JIT-компиляции (Just-In-Time). Это значит, что код компилируется прямо во время выполнения, а не заранее.
  <br>
  Ниже представлен подробный конспект архитектуры и жизненного цикла выполнения кода в V8.
</p>

<h3>1. Парсинг (От текста к структуре)</h3>

<p class="info info--blue">
  Чтобы ускорить загрузку страницы, движок не парсит функции, которые пока не были вызваны. <strong>Pre-parser</strong> просто пробегается по ним, проверяя базовый синтаксис, а полное AST строится только в момент вызова функции.
</p>

<p>
  Прежде чем код начнет выполняться, движок должен его прочитать и понять. Этот этап делится на две фазы:
</p>
<p>
  1.1. <strong>Лексический анализ (Tokenization):</strong> Код разбивается на мельчайшие значимые единицы — токены (ключевые слова, операторы, скобки).
</p>

<code class="code">
  const a = 5;
  // Токены: const, a, =, 5, ;

  function greet(name) {
    return `Hello, ${name}!`;
  }
  // Токены: function, greet, (, name, ), {, return, `Hello, ${name}!`, }
</code>

<p>
  1.2. <strong>Синтаксический анализ (Parsing):</strong> Из токенов строится AST (Abstract Syntax Tree / Абстрактное синтаксическое дерево) — древовидное представление логики кода.
</p>

<h3>2. Интерпретация (Быстрый старт программы)</h3>

<p class="info info--blue">
  <strong>Интерпретация</strong> — это процесс пошагового выполнения байт-кода без его предварительной компиляции в нативный машинный код процессора.
  <br><br>
  Главная цель этого этапа — максимально быстрый запуск приложения. Движку не нужно тратить драгоценное время на глубокую компиляцию и оптимизацию всего скрипта, чтобы страница начала реагировать на действия пользователя.
  <br><br>
  Если провести аналогию, то процесс интерпретации — это синхронный переводчик. Он стоит рядом с тобой и переводит каждую твою фразу на китайский прямо в момент разговора.
</p>

<p>
  Представь, что браузер загрузил мегабайт JavaScript-кода. Если бы он пытался скомпилировать его целиком до запуска, пользователь видел бы белый экран несколько секунд.
</p>

<p class="info info--blue">
  <strong>Компиляция</strong> — это перевод программы с языка программирования, понятного человеку (например, JS, Rust, C++), на язык, который понимает процессор компьютера (машинный код / двоичный код / assembly), состоящий из нулей и единиц.
  <br><br>
  Продолжая аналогию, то процесс компиляции — это переводчик, который заранее перевел все фразы, чтобы потом можно было быстро их озвучить.
</p>

<p>
  Чтобы избежать подобной задержки, движок использует следующий подход:
</p>
<ul>
  <li>
    <strong>Генерация байт-кода:</strong> Специальный компонент движка (в V8 это интерпретатор <strong>Ignition</strong>) проходит по абстрактному синтаксическому дереву (AST) и преобразует его в промежуточный формат — байт-код.
  </li>
  <li>
    <strong>Ленивая инициализация (Lazy Evaluation):</strong> Байт-код не генерируется для всего файла сразу. Если в коде есть функции, которые пока не вызывались, Ignition просто запомнит их, но не будет обрабатывать. Код для них сгенерируется ровно в момент их первого вызова, что колоссально экономит ресурсы на старте.
  </li>
  <li>
    <strong>Непосредственное выполнение:</strong> Сгенерированный байт-код начинает выполняться мгновенно. Да, пошаговое чтение инструкций работает медленнее, чем оптимизированный машинный код, но зато мы полностью пропускаем фазу тяжелой компиляции при загрузке.
  </li>
</ul>

<p>
  Давай посмотрим, как это выглядит на практике. Исходный код превращается в набор простых пошаговых инструкций, которые легко читает интерпретатор:
</p>

<code class="code">
  // Наш исходный JavaScript-код
  function add(a, b) {
    return a + b;
  }

  /* Примерный вид сгенерированного байт-кода в V8 (Ignition).
    Интерпретатор выполняет эти инструкции последовательно:
  */
  LdaSmi a   // Загрузить значение переменной 'a' в регистр-аккумулятор
  Star r0    // Сохранить текущее значение аккумулятора во временный регистр 'r0'
  LdaSmi b   // Загрузить значение переменной 'b' в регистр-аккумулятор
  Add r0     // Сложить текущее значение аккумулятора (b) с тем, что лежит в 'r0' (a)
  Return     // Вернуть итоговый результат из аккумулятора
</code>

<h3>3. Оптимизация (JIT-компиляция)</h3>

<p>
  Выполняя байт-код, движок параллельно анализирует его выполнение и собирает статистику. На основе этих данных он может принять решение о том, что определенные участки кода стоит оптимизировать, скомпилировав их в машинный код.
</p>

<p>
  <strong>Профайлер (Profiler / Monitor)</strong> - один из компонентов движка. Он считает частоту вызовов функций. Функции, которые выполняются многократно (например, внутри циклов), помечаются как "горячие" (hot).
</p>

<p>
  Сам интерпретатор с помощью механизма <strong>Inline Caches (IC)</strong> отслеживает, какие типы данных передаются в функции. Например, если функция всегда получает числа, это важная информация для оптимизации.
</p>

<p>
  После этого, основываясь на собранной информации, движок делает предположения - это называется <strong>спекулятивная оптимизация</strong>.
</p>

<p>
  Если <strong>Profiler</strong> видел, что функция всегда получает на вход только числа, <strong>TurboFan</strong> скомпилирует машинный код, заточенный исключительно под работу с числами, отбросив проверки типов.
</p>

<p>
  Оптимизированный код заменяет интерпретируемый байт-код, и дальнейшие вызовы функции выполняются уже с высокой производительностью.
</p>

<p>
  Однако, если в будущем функция начнет получать другие типы данных (например, строки), оптимизированный код может стать неэффективным. В этом случае движок может "деоптимизировать" функцию, вернувшись к интерпретируемому байт-коду. Это называется <strong>Деоптимизация (Bailout / Deoptimization)</strong>.
  <br>
  Деоптимизация — это очень ресурсоемкая операция. Поэтому движки стараются избегать ее, используя различные эвристики для определения, когда оптимизировать код.
</p>

<p class="info info--blue">
  Важно понимать, что оптимизация — это непрерывный процесс. Движок постоянно собирает данные и может повторно оптимизировать код, если ситуация изменится.
</p>

<h3>Как писать код, который движок сможет оптимизировать</h3>

<p>
  Как разработчик, ты можешь помочь JIT-компилятору не делать лишней работы и не уходить в деоптимизацию.
</p>

<p>
  <strong>Соблюдай мономорфизм:</strong> Старайся передавать в конкретную функцию аргументы одного и того же типа.
</p>

<p>
  <strong>Зафиксируй форму объектов (Скрытые классы):</strong> Движки создают под капотом "скрытые классы" (Hidden Classes / Shapes) для объектов с одинаковым набором ключей. Всегда инициализируй свойства объекта в конструкторе и строго в одном и том же порядке.
</p>

<p>
  <strong>Избегай оператора <code>delete</code>:</strong> Удаление свойств из объекта "ломает" его скрытый класс, заставляя движок создавать новый и отбрасывать предыдущие оптимизации доступа к памяти.
</p>

<p>
  <strong>Не меняй типы элементов массивов:</strong> Если массив начал свой жизненный цикл как массив чисел (например, <code>[1, 2, 3]</code>), не пушь в него строки или объекты. Это заставит движок перестроить массив в памяти по менее эффективному шаблону.
</p>
