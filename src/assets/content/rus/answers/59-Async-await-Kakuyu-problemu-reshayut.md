
<h3>Эволюция асинхронного JS в ES8</h3>

<p>
  <code>async</code> и <code>await</code> — это логичное продолжение эволюции асинхронного JavaScript,
  появившееся в стандарте <b>ES2017 (ES8)</b>.
  <br>
  Строго говоря, это не какая-то новая механика, а
  <b>синтаксический сахар</b> над уже знакомыми нам Промисами.
  <br>
  Под капотом они работают точно так же.
</p>

<h3>Зачем нужен async/await? Какую проблему решает?</h3>
<p>
  Промисы избавили нас от <b>ада колбэков</b>, но породили новую, пусть и меньшую, проблему — визуальный шум и
  сложность ветвления в цепочках <code>.then()</code>.
</p>

<p>
  Когда у вас есть простая последовательность, <code>.then()</code> читается отлично. Но если вам нужно передать
  переменную из первого <code>.then()</code> в третий, или добавить сложное условие <code>if/else</code> посередине цепочки, код
  начинает обрастать вложенностью, лишними функциями и становится трудным для восприятия.
</p>

<p>
  <code>async</code>/<code>await</code> решает эту проблему радикально: он позволяет писать асинхронный
  код так, как будто он синхронный (линейный, читающийся строго сверху вниз).
</p>

<h3>Как работают эти два слова</h3>
<ol>
  <li>
    <b>async (Асинхронная функция)</b><br>
    Слово <code>async</code> ставится перед объявлением функции. У него есть ровно одна задача: гарантировать,
    что функция всегда вернет Промис. Даже если вы напишете <code>return 1;</code> внутри async-функции, движок
    JavaScript неявно обернет эту единицу в успешно завершенный промис (<code>Promise.resolve(1)</code>).
  </li>

  <li>
    <b>await (Ожидание результата)</b><br>
    Слово <code>await</code> может использоваться только внутри async-функции. Оно заставляет JavaScript ждать,
    пока промис справа от него не завершится, и возвращает его результат. Если промис завершится с ошибкой,
    будет выброшено исключение, которое можно поймать через <code>try/catch</code>.
  </li>
</ol>

<code class="code">
  async function getUserData() {
    try {
      // Код читается как обычная книга — сверху вниз
      const response = await fetch('/api/user');
      const user = await response.json();
      console.log('Пользователь:', user.name);

      const postsResponse = await fetch('/api/posts/userId');
      const posts = await postsResponse.json();
      console.log('Посты:', posts);

    } catch (error) {
      // Обработка ошибок теперь делается классическим try...catch
      console.error('Ошибка:', error);
    }
  }
</code>

<h3>try...catch...finally</h3>

<p>
  С <code>async/await</code> обработка ошибок стала намного проще и интуитивнее. Вместо того, чтобы цеплять <code>.catch()</code>
  к каждому промису,
  мы можем просто обернуть весь код в блок <code>try</code> и поймать любые ошибки в одном месте через <code>catch</code>.
  <br>
  Это делает код чище и позволяет легко управлять ошибками, особенно когда у нас есть несколько последовательных асинхронных операций.
  <br>

  Метод <code>finally</code> также работает с async/await, позволяя выполнять код, который должен быть выполнен в любом случае,
  независимо от того, произошла ошибка или нет (например, очистка ресурсов, скрытие загрузочного индикатора и т.д.).
</p>

<h3>Про Node.js</h3>

<p>
  В Node.js поддержка async/await появилась в версии 7.6 (2017 год),
  но сначала была экспериментальной и требовала запуска с флагом <code>--harmony-async-await</code>.
  <br>
  Сейчас же в Node.js (начиная с версии 14.8.0 без флагов) <code>await</code> можно смело использовать прямо в корне файла.
</p>

<p class="info info--orange">
  Top-level await работает только внутри ES-модулей (ESM). В классической системе CommonJS
  (где используется require и module.exports) он работать не будет — движок выдаст синтаксическую ошибку.
</p>
