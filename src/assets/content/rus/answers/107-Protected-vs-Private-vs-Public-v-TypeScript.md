<h3>Главное правило (TL;DR)</h3>
<p>По умолчанию всегда делай всё <code>private</code>. Открывай доступ (до <code>protected</code> или <code>public</code>) только тогда, когда в этом появляется реальная архитектурная необходимость. Это убережет от "спагетти-кода", когда компоненты лезут во внутреннее состояние друг друга.</p>

<h3>1. public (Публичный)</h3>
<p>Доступен везде. Внутри класса, в классах-наследниках, в экземплярах класса снаружи.</p>

<p class="info">В TypeScript, если ты не написал модификатор, свойство или метод автоматически становится <code>public</code>.</p>

<p>Когда использовать:</p>

<p> - Для формирования публичного API твоего класса или сервиса. Это те методы, ради которых класс и создавался.</p>

<p> - Для методов и свойств, которые вызываются из HTML-шаблонов (особенно актуально для компонентов Angular или Vue 2, где шаблон по сути находится "снаружи" класса компонента).</p>

<code class="code">
  class UserService {
    // Этот метод нужен внешнему миру, чтобы получить данные
    public getUser(): User { ... }
  }
</code>

<h3>2. private (Приватный)</h3>

<p>Доступен только внутри того класса, где он объявлен. Даже классы-наследники (через <code>extends</code>) не увидят <code>private</code> методы своего родителя.</p>

<p>Когда использовать:</p>

<p> - Для внутреннего состояния (state), которое никто не должен менять напрямую в обход твоей логики.</p>

<p> - Для вспомогательных методов-хелперов, которые выполняют грязную работу внутри класса, но снаружи абсолютно бесполезны.</p>

<p> - Для инжектирования зависимостей (DI) в конструкторе сервисов, если эти зависимости нужны только самому сервису.</p>

<code class="code">
  class UserService {
    // Внутреннее состояние, никто не должен трогать напрямую
    private users: User[] = [];

    // Вспомогательный метод, бесполезный снаружи
    private formatUserData(data: any): User { ... }

    // DI, если HttpClient нужен только этому сервису
    constructor(private http: HttpClient) { ... }
  }
</code>

<p class="info info--orange">
  Примечание: В современном JS/TS также есть хардкорный приватный модификатор <code>#</code> (например, <code>#balance</code>), который реально скрывает данные в рантайме, в отличие от слова <code>private</code>, которое работает только на этапе компиляции TypeScript.
</p>

<h3>3. protected (Защищенный)</h3>

<p>Доступен внутри самого класса и во всех его классах-наследниках. Снаружи (у экземпляра) доступа нет.</p>

<p>Когда использовать:</p>

<p> - При проектировании базовых/абстрактных классов.</p>

<p>
   - Когда у тебя есть общая логика для группы компонентов (например, базовый класс табличного компонента с пагинацией), и дочерним компонентам нужен доступ к методам родителя, чтобы переопределить их или использовать, но наружу этот функционал торчать не должен.
</p>

<p> - Когда ты хочешь, чтобы наследники могли расширять функциональность базового класса, но не хочешь открывать эти методы для внешнего мира.</p>

<code class="code">
  abstract class BaseTableComponent {
    // Доступно наследникам, но закрыто от внешнего мира
    protected currentPage: number = 1;

    protected abstract loadData(): void;
  }

  class UsersTableComponent extends BaseTableComponent {
    public nextPage() {
      this.currentPage++; // Имеем доступ к protected свойству
      this.loadData();
    }

    protected loadData(): void {
      // Реализация загрузки
    }
  }
</code>

<p class="info">
  Важно: Не стоит злоупотреблять <code>protected</code>, превращая его в "почти public". Если тебе кажется, что многие методы должны быть <code>protected</code>, возможно, стоит пересмотреть архитектуру и выделить эти методы в отдельный сервис или утилиту.
</p>

<h3>Итог - когда использовать каждый из этих модификаторов доступа?</h3>

<p>
  <code>protected</code> нужен только для того, чтобы разграничить доступ. Чтобы дать возможность использовать методы в наследниках (ведь <code>private</code> там не доступны) и закрыть доступ к вызову методов снаружи (как для <code>public</code>).
  <br />
  Если тебе нужно, чтобы метод был доступен и для наследников, и для внешнего мира, то он должен быть <code>public</code>.
  <br />
  Если же он нужен только для наследников, то <code>protected</code>.
  <br />
  Если же он нужен только внутри класса, то <code>private</code>.
</p>
