<h3>Function Declaration (Объявление функции)</h3>

<p>Это классический способ создания функции. Она объявляется отдельной конструкцией в основном потоке кода.</p>

<code class="code">
  function greet(name) {
    return `Hello, ${name}!`;
  }
</code>

<h4>Ключевые особенности:</h4>

<p><strong>Имя обязательно:</strong> Функция должна иметь имя (greet), чтобы парсер JS понял, что это именно Declaration.</p>

<p><strong>Hoisting (Поднятие):</strong> Это самая важная механика. Движок JavaScript сканирует код до его выполнения и «поднимает» все Function Declarations на самый верх их области видимости (глобальной или функциональной).</p>

<p><strong>Использование:</strong> Идеально подходит для создания глобальных утилит или основных логических блоков, которые должны быть доступны в любом месте файла.</p>

<code class="code">
  // Функция вызывается ДО ее объявления в коде — и это работает.
  const total = getDiscount(100);

  function getDiscount(price) {
    return price * 0.9;
  }
</code>

<p class="info info--orange">У стрелочных функций нет возможности быть Function Declarations - они всегда являются Function Expressions.</p>

<h3>Function Expression (Функциональное выражение)</h3>

<p>
  Это способ создания функции, при котором функция присваивается переменной. Она может быть анонимной (без имени) или иметь имя (Named Function Expression).
  <br>
  JS воспринимает такую функцию просто как значение.
</p>

<code class="code">
  // Анонимное функциональное выражение
  const greet = function(name) {
    return `Hello, ${name}!`;
  };
</code>

<h4>Ключевые особенности:</h4>

<p><strong>Имя необязательно:</strong> Обычно они анонимны (как в примере выше), но могут быть и именованными (Named Function Expression).</p>

<p><strong>Отсутствие Hoisting:</strong> Функция создается ровно в тот момент, когда поток выполнения (execution flow) доходит до этой строки. Ее нельзя вызвать до инициализации.</p>

<p><strong>Использование:</strong> Замыкания, коллбеки, передача функций как аргументов, методы объектов.</p>

<h3>Пример ошибки (TDZ — Temporal Dead Zone):</h3>

<code class="code">
  // ReferenceError: Cannot access 'getDiscount' before initialization
  const total = getDiscount(100);

  const getDiscount = function(price) {
    return price * 0.9;
  };
</code>

<h3>Named Function Expression (NFE)</h3>

<p>Разновидность Function Expression, где у функции есть локальное имя. Оно доступно только внутри самой этой функции. Это критически полезно для рекурсии и понятных стектрейсов при дебаггинге.</p>

<code class="code">
  const factorial = function calc(n) {
    if (n <= 1) return 1;
    return n * calc(n - 1); // Обращение к себе же по внутреннему имени
  };

  console.log(factorial(5)); // 120
  console.log(calc(5)); // ReferenceError: calc is not defined
</code>


<h3>Важные нюансы и рекомендации</h3>

<p><strong>Консистентность:</strong> Старайся не смешивать стили без необходимости. Для вспомогательных функций в корне файла используй Declaration (чтобы они не захламляли верхнюю часть файла, их можно убрать вниз). Для логики, зависящей от условий, или коллбеков — Expression.</p>

<p><strong>Стрелочные функции (Arrow Functions):</strong> Помни, что в современном фронтенде многие Function Expressions заменяются на стрелочные функции (<code>const sum = (a, b) => a + b</code>). Они тоже являются выражениями и не всплывают (no hoisting), но дополнительно не имеют собственного контекста <code>this</code> (что часто спасает при передаче коллбеков в компонентах).</p>

<p><strong>Строгий режим (<code>'use strict'</code>):</strong> В строгом режиме Function Declaration имеет блочную область видимости. Если объявить ее внутри <code>if {}</code>, снаружи этого блока она будет недоступна.</p>

<p><strong>Рекурсия:</strong> Если функция должна вызывать себя же, и ты используешь Function Expression, лучше сделать ее Named Function Expression, чтобы избежать проблем с доступом к функции внутри самой себя.</p>

<p><strong>Производительность:</strong> В большинстве случаев разницы в производительности между этими способами нет. Выбирай стиль, который лучше подходит для конкретной задачи и улучшает читаемость кода.</p>

<p class="info info--blue">
  Хотя существует миф о том, что большое количество Function Declarations может замедлять выполнение кода, на практике современные движки JavaScript оптимизируют их выполнение, и разница в производительности минимальна.
  <br>
  Разница во времени инициализации между тысячей Function Declarations и тысячей Function Expressions ничтожна и не поддается адекватному измерению в реальных условиях.
</p>

<h3>Под капотом: Движок и Спецификация ECMAScript</h3>

<p>Согласно спецификации ECMAScript, когда JavaScript-движок (например, V8 в Chrome) выполняет код, он проходит две фазы.</p>

<h4>Фаза 1: Creation Phase (Фаза создания / компиляции)</h4>

<p>До того как будет выполнена хотя бы одна строчка кода, движок парсит скрипт (переводит его в токены), строит AST (абстрактное синтаксическое дерево) и выделяет память под переменные и функции. Это называется инициализацией Lexical Environment (лексического окружения).</p>

<p>Во время этой фазы все Function Declarations обрабатываются и их имена добавляются в область видимости. Это означает, что даже если функция объявлена внизу файла, она будет доступна вверху.</p>

<p>Function Expressions, с другой стороны, не обрабатываются на этом этапе. Движок видит только объявление переменной (например, <code>const sum = ...</code> или <code>var sum = ...</code>).</p>

<h4>Фаза 2: Execution Phase (Фаза выполнения)</h4>
<p>Движок начинает выполнять код строчка за строчкой (сверху вниз).</p>

<p><strong>FD:</strong> Движок просто пропускает строки с Function Declaration, так как они уже были обработаны и загружены в память на первой фазе.</p>

<p><strong>FE:</strong> Когда поток выполнения (Execution flow) доходит до строки с <code>const sum = function() {}</code>, происходит операция вычисления выражения (Evaluation). Только в эту микросекунду движок создает Function Object в куче (heap) и сохраняет ссылку на него в переменную <code>sum</code>.</p>

<h3>Разница на уровне грамматики языка (AST)</h3>
<p>В спецификации ECMAScript это две принципиально разные синтаксические конструкции:</p>

<p><strong>Statement (Инструкция / Утверждение):</strong> Function Declaration — это statement. Она выполняет действие (регистрирует функцию в окружении), но не возвращает значение. Ее нельзя передать как аргумент или присвоить переменной напрямую.</p>

<p><strong>Expression (Выражение):</strong> Function Expression — это часть выражения. Выражение в JS — это любой кусок кода, который возвращает значение. Движок вычисляет FE и возвращает значение — саму функцию (объект в памяти), которую мы затем можем присвоить переменной, передать в <code>setTimeout</code> или использовать в замыкании.</p>

<p>Классический сеньорский пример на понимание грамматики:</p>

<code class="code">
  // Это Function Declaration. Вызовет синтаксическую ошибку (SyntaxError),
  // так как statement не может быть вызван на месте, а парсер ожидает имя функции.
  function() {
    console.log('test');
  }();
  // SyntaxError: Function statements require a function name

  // Оборачиваем в скобки (Grouping Operator).
  // Скобки в JS могут содержать только выражения (Expressions).
  // Парсер мгновенно превращает FD в Function Expression.
  // Теперь это IIFE (Immediately Invoked Function Expression), и код работает.
  (function() {
    console.log('test');
  })();
</code>

<h3>Миф: Function Declaration тормозит инициализацию (Performance)</h3>
<p>Суть мифа: Так как все Function Declarations всплывают (Hoisting) и инициализируются в памяти на фазе создания контекста (до старта выполнения кода), огромное их количество якобы сильно замедлит старт приложения.</p>

<h4>Как работает на самом деле (Lazy Parsing):</h4>
<p>Современные JS-движки (V8, SpiderMonkey) невероятно оптимизированы. Чтобы не тратить время и память на старте, они используют механизм ленивого парсинга (Lazy / Pre-parsing).</p>

<p><strong>Pre-parsing на старте:</strong> Когда движок сканирует код и находит Function Declaration, он не компилирует ее тело целиком. Он делает лишь быстрый проход: проверяет базовый синтаксис и регистрирует имя функции в лексическом окружении. Это занимает доли миллисекунд.</p>

<p><strong>Lazy Compilation:</strong> Полноценный парсинг тела функции, построение полного AST и генерация байт-кода происходят только в момент первого вызова этой функции. Если функция никогда не будет вызвана, движок так и не потратит ресурсы на ее глубокую компиляцию.</p>
