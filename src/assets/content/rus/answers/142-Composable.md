<h3>Что такое Composable?</h3>

<p>
  <span class="accent">Composable</span> - это функция, которая использует Composition API (<code>ref</code>, <code>reactive</code>, <code>computed</code>, <code>watch</code>, хуки жизненного цикла и т.д.) для инкапсуляции и повторного использования реактивного состояния и логики в компонентах Vue.js.
  <br>
  <strong>Composable</strong> позволяет разработчикам создавать абстрактные функции, которые могут быть легко интегрированы в различные компоненты, обеспечивая более чистый и модульный код.
</p>

<p>
  <strong>Composable</strong> - это не встроенная концепция Vue, а скорее паттерн проектирования / соглашение, который разработчики могут использовать для организации и повторного использования логики в своих приложениях.
</p>

<p class="info info--orange">
  Существует строгое соглашение сообщества относительно нейминга <strong>composable</strong> - использование use-нейминга (например, <code>useFetch</code>, <code>useAuth</code>). Это соглашение помогает сразу идентифицировать функции, которые являются composables, и отличать их от обычных утилит или сервисов.
</p>

<h3>В чем отличие Composable от обычной утилиты (utils)?</h3>

<p>
  Важно понимать разницу между функциями без состояния (stateless) и функциями с состоянием (stateful).
</p>

<p>
  <strong>Обычная утилита (utility)</strong> — это чистая функция, которая принимает аргументы, выполняет логику (например, форматирование даты или сложные математические вычисления) и возвращает результат. Она ничего не знает о Vue и не хранит в себе изменяемых данных.
</p>

<p>
  <strong>Composable</strong> — это функция, которая использует возможности Vue (Composition API) для управления реактивным состоянием или побочными эффектами (подписки на события, таймеры через хуки жизненного цикла). Она хранит в себе переиспользуемую логику, которая может включать в себя реактивные данные и взаимодействовать с жизненным циклом компонента.
</p>

<p class="info info--blue">
  По сути, главное отличие кроется в наличии реактивности: как только обычная функция-утилита начинает инкапсулировать внутри себя состояние (добавили <code>ref</code> или <code>reactive</code>) или использовать хуки (<code>onMounted</code>), она превращается в <strong>composable</strong>.
</p>

<h5>Пример НЕ composable:</h5>

<p>
  По официальной документации Vue, Composable — это функция, которая инкапсулирует и переиспользует stateful logic (логику с состоянием).
  <br>
  Данный пример не является классическим композаблом по трем причинам:
  <br>
  - Она не создает и не инкапсулирует внутри себя независимое состояние (нет своих <code>ref</code> или <code>reactive</code>).
  <br>
  - Она не возвращает никаких реактивных данных для использования в компоненте.
  <br>
  - Нарушено базовое соглашение об именовании - функция не начинается с приставки use (должно быть что-то вроде useLogger).
</p>

<code class="code">
  function logCount(count) {
    watch(count, (val) => console.log(val));
  }
</code>

<h5>Пример composable:</h5>

<p>
  Эта функция использует <code>ref</code> для создания реактивного состояния и возвращает его, что позволяет компоненту, который вызывает эту функцию, реагировать на изменения этого состояния. Это делает её composable.
</p>

<code class="code">
  function useCounter() {
    const count = ref(0);

    const increment = () => count.value++;

    return { count, increment };
  }

</code>

<h3>Золотые правила написания Composables</h3>

<p>
  <strong>Нейминг:</strong> Имя функции всегда должно начинаться с use (например, <code>useFetch</code>, <code>useMouse</code>, <code>useAuth</code>).
</p>

<p><strong>Инкапсуляция:</strong> Вся логика, связанная с конкретной задачей, должна быть внутри одной функции.</p>

<p>
  <strong>Возвращаемое значение:</strong> Возвращай простой JavaScript-объект, содержащий <code>ref</code> (а не <code>reactive</code> объект). Это позволяет безопасно деструктурировать результат в компоненте, не теряя реактивность
</p>

<p>
  <strong>Работа с реактивными аргументами:</strong> Хороший композабл должен уметь работать с реактивными аргументами. Для "распаковки" переданных аргументов внутри функции стоит использовать <code>unref()</code> (или <code>toValue()</code> в Vue 3.3+). Это позволяет композаблу реагировать на изменения входных данных.
</p>

<p>
  <strong>Уборка за собой:</strong> Если композабл создает слушатели событий (event listeners) или таймеры, он обязан очистить их в хуке <code>onUnmounted</code>.
</p>

<h3>Базовый шаблон (на примере отслеживания мыши)</h3>

<code class="code">
  import { ref, onMounted, onUnmounted } from 'vue';

  export function useMouse() {
    // 1. Инициализация реактивного состояния
    const x = ref(0);
    const y = ref(0);

    // 2. Логика обновления состояния
    function update(event) {
      x.value = event.pageX;
      y.value = event.pageY;
    }

    // 3. Подписка на события при монтировании
    onMounted(() => window.addEventListener('mousemove', update));

    // 4. Очистка при размонтировании (КРИТИЧЕСКИ ВАЖНО)
    onUnmounted(() => window.removeEventListener('mousemove', update));

    // 5. Возврат ref-ов
    return { x, y };
  }
</code>

<h3>Локальное vs Глобальное состояние</h3>

<p>
  <strong>Локальное (по умолчанию):</strong> Состояние (<code>ref</code>) объявляется внутри функции. Каждый компонент, вызывающий useSomething(), получит свою независимую копию состояния.
</p>

<p>
  <strong>Глобальное (Shared):</strong> Состояние объявляется снаружи функции, в замыкании модуля. Все компоненты, вызывающие композабл, будут делить одно и то же реактивное состояние (мини-замена Vuex/Pinia).
</p>

<code class="code">
  import { ref } from 'vue'

  // Глобальное состояние (одно на все приложение)
  const globalCount = ref(0)

  export function useCounter() {
    // Локальное состояние (новое для каждого компонента)
    const localCount = ref(0)

    return { globalCount, localCount }
  }
</code>

<h3>Композиция внутри композиции</h3>

<p>
  Композаблы могут вызывать другие композаблы. Например, useFetch может использовать внутри себя useEventListener для отслеживания статуса сети (online/offline) перед запросом.
</p>

<h3>Когда использовать Composables?</h3>
<p>
  <strong>Повторяющаяся логика:</strong> Когда у тебя есть логика, которая повторяется в нескольких компонентах (например, работа с формами, аутентификация, взаимодействие с API), композаблы позволяют инкапсулировать эту логику и легко повторно использовать её.
</p>

<h3>Какие преимущества дает использование Composables?</h3>
<p>
  <strong>Повторное использование:</strong> Композаблы позволяют легко повторно использовать логику в разных компонентах, что уменьшает дублирование кода.
</p>

<p>
  <strong>Чистота компонентов:</strong> Вынося сложную логику в композаблы, компоненты становятся чище и легче читаемыми, так как они фокусируются только на представлении.
</p>

<p>
  <strong>Гибкость:</strong> Композаблы могут быть легко комбинированы друг с другом, что позволяет создавать сложные функции из более простых.
</p>

<h3>Когда НЕ стоит использовать Composables?</h3>
<p>
  <strong>Простая логика:</strong> Если логика очень простая и используется только в одном месте, создание композабла может быть избыточным.
</p>

<p>
  <strong>Узкоспециализированные функции:</strong> Если функция предназначена для очень специфической задачи, которая не имеет смысла быть повторно используемой, она может не нуждаться в статусе композабла.
</p>

<h3>Ключевые отличия Composable от mixins и HOCs</h3>
<p>
  <strong>Mixins:</strong> Mixins могут создавать проблемы с конфликтами имен и неявным происхождением методов или зависимостей (Implicit dependencies), тогда как composables возвращают явные объекты и не имеют таких проблем.
</p>
<p>
  <strong>HOCs (Higher-Order Components):</strong> HOCs оборачивают компоненты и могут создавать сложные иерархии, тогда как composables просто возвращают объекты с состоянием и логикой, которые можно использовать внутри компонентов без оборачивания.
</p>
