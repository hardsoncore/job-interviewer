<h4>Навіщо потрібні?</h4>

<p><span class="accent">Лінтери та форматери коду</span> — це інструменти, які допомагають підтримувати єдиний стиль написання коду та запобігати поширеним помилкам. Вони аналізують ваш код на наявність проблем і допомагають підтримувати його відповідно до заданих правил.</p>

<p>Налаштування єдиних стандартів та автоматизація перевірок — це те, що відрізняє хаотичний набір скриптів від дорослого продукту, який легко підтримувати. Коли над кодовою базою працює команда, інструменти автоматизації беруть на себе рутину, дозволяючи розробникам обговорювати на код-рев'ю архітектуру та бізнес-логіку, а не пропущені крапки з комою.</p>

<h4>1. Форматування коду: Prettier</h4>
<p><span class="accent">Prettier</span> — це форматер коду. Він бере ваш код, ігнорує початкове форматування і переписує його наново за заздалегідь заданими правилами (довжина рядка, лапки, відступи).</p>

<p><strong>Яку проблему вирішує:</strong> Повністю усуває суперечки про стиль коду в команді. Код завжди виглядає однаково, незалежно від того, хто його писав. Це також зводить до мінімуму безглузді merge conflicts через різну кількість пробілів.</p>

<p><strong>Особливість:</strong> Prettier форматує код, але не перевіряє його логіку.</p>

<h4>2. Лінтинг коду: ESLint та Stylelint</h4>
<p><span class="accent">ESLint</span> — це інструмент для статичного аналізу коду. Він перевіряє ваш код на наявність потенційних помилок, поганих практик та порушень стилю. На відміну від Prettier, ESLint може бути налаштований на перевірку логіки коду, використання змінних, правил іменування та багато іншого.</p>

<p><strong>Яку проблему вирішує:</strong> Допомагає виявити помилки та потенційні проблеми в коді до його виконання. Це може включати невикористані змінні, неправильне використання функцій або навіть потенційні вразливості безпеки.</p>
<p><strong>Особливість:</strong> ESLint може бути налаштований на виконання певних правил, які можуть бути специфічними для вашого проєкту або команди.</p>

<p><span class="accent">Stylelint</span> — це аналог ESLint, але для CSS та інших стилів. Він допомагає підтримувати єдиний стиль написання CSS, SCSS та інших препроцесорів.</p>

<h4>3. Базові налаштування редактора: .editorconfig</h4>

<p><span class="accent">.editorconfig</span> — це простий конфігураційний файл, який лежить у корені проєкту і розпізнається більшістю сучасних IDE (VS Code, WebStorm тощо).</p>

<p><strong>Яку проблему вирішує:</strong> Синхронізує базові налаштування текстових редакторів у всіх членів команди до того, як у справу вступлять лінтери.</p>

<p><strong>Що контролює:</strong> Розмір відступу (tab vs space), тип закінчення рядків (LF або CRLF — критично важливо, якщо в команді є користувачі Windows, macOS та Linux), обов'язковий порожній рядок у кінці файлу.</p>

<h4>4. Контроль комітів: Husky + lint-staged</h4>
<p>Налаштувати лінтери замало — потрібно змусити команду їх використовувати. Якщо хтось забув увімкнути автоформатування в редакторі, брудний код потрапить до репозиторію.</p>

<p><strong>Husky:</strong> Інструмент для зручної роботи з Git Hooks. Дозволяє запускати скрипти на певні дії в git (наприклад, перед комітом — pre-commit, або перед пушем — pre-push).</p>

<p><strong>lint-staged:</strong> Утиліта, яка запускає лінтери тільки для тих файлів, які додані в індекс (staged) для поточного коміту.</p>

<p><strong>Як це працює разом:</strong> При спробі зробити git commit, Husky перехоплює команду і запускає lint-staged. Той проганяє змінені файли через Prettier та ESLint. Якщо є помилки — коміт скасовується.</p>

<h4>5. Стандарти повідомлень комітів: Commitlint</h4>

<p>Одноманітний код — це добре, але читабельна історія Git — ще краще.</p>

<p><strong>Яку проблему вирішує:</strong> Забороняє коміти з повідомленнями на кшталт "fix", "wip" або "asdfgh".</p>

<p><strong>Як працює:</strong> Змушує дотримуватися конвенції (найчастіше Conventional Commits). Коміт має виглядати як <code>feat(auth): add login modal</code> або <code>fix(api): resolve timeout issue</code>. Це дозволяє пізніше автоматично генерувати <code>CHANGELOG.md</code> і легко орієнтуватися в історії релізів.</p>
