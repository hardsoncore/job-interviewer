<p>
  Сучасні JS-рушії (розглянемо на прикладі найпопулярнішого — V8 від Google) використовують модель JIT-компіляції (Just-In-Time). Це означає, що код компілюється безпосередньо під час виконання, а не заздалегідь.
  <br>
  Нижче наведено детальний конспект архітектури та життєвого циклу виконання коду у V8.
</p>

<h3>1. Парсинг (Від тексту до структури)</h3>

<p class="info info--blue">
  Щоб прискорити завантаження сторінки, рушій не парсить функції, які ще не були викликані. <strong>Pre-parser</strong> просто пробігається по них, перевіряючи базовий синтаксис, а повне AST будується лише в момент виклику функції.
</p>

<p>
  Перш ніж код почне виконуватися, рушій повинен його прочитати та зрозуміти. Цей етап поділяється на дві фази:
</p>
<p>
  1.1. <strong>Лексичний аналіз (Tokenization):</strong> Код розбивається на найдрібніші значущі одиниці — токени (ключові слова, оператори, дужки).
</p>

<code class="code">
  const a = 5;
  // Токени: const, a, =, 5, ;

  function greet(name) {
    return `Hello, ${name}!`;
  }
  // Токени: function, greet, (, name, ), {, return, `Hello, ${name}!`, }
</code>

<p>
  1.2. <strong>Синтаксичний аналіз (Parsing):</strong> З токенів будується AST (Abstract Syntax Tree / Абстрактне синтаксичне дерево) — деревоподібне подання логіки коду.
</p>

<h3>2. Інтерпретація (Швидкий старт програми)</h3>

<p class="info info--blue">
  <strong>Інтерпретація</strong> — це процес покрокового виконання байт-коду без його попередньої компіляції в нативний машинний код процесора.
  <br><br>
  Головна мета цього етапу — максимально швидкий запуск додатка. Рушію не потрібно витрачати дорогоцінний час на глибоку компіляцію та оптимізацію всього скрипта, щоб сторінка почала реагувати на дії користувача.
</p>

<p>
  Уявімо, що браузер завантажив мегабайт JavaScript-коду. Якби він намагався скомпілювати його цілком до запуску, користувач бачив би білий екран кілька секунд.
</p>

<p class="info info--blue">
  <strong>Компіляція</strong> — це переклад програми з мови програмування, зрозумілої людині (наприклад, JS, Rust, C++), на мову, яку розуміє процесор комп'ютера (машинний код, що складається з нулів і одиниць).
</p>

<p>
  Щоб уникнути подібної затримки, рушій використовує наступний підхід:
</p>
<ul>
  <li>
    <strong>Генерація байт-коду:</strong> Спеціальний компонент рушія (у V8 це інтерпретатор <strong>Ignition</strong>) проходить по абстрактному синтаксичному дереву (AST) і перетворює його в проміжний формат — байт-код.
  </li>
  <li>
    <strong>Лінива ініціалізація (Lazy Evaluation):</strong> Байт-код не генерується для всього файлу одразу. Якщо в коді є функції, які поки не викликалися, Ignition просто запам'ятовує їх, але не обробляє. Код для них буде згенерований саме в момент їх першого виклику, що значно економить ресурси на старті.
  </li>
  <li>
    <strong>Непосереднє виконання:</strong> Згенерований байт-код починає виконуватися миттєво. Так, покрокове читання інструкцій працює повільніше, ніж оптимізований машинний код, але ми повністю пропускаємо фазу важкої компіляції при завантаженні.
  </li>
</ul>

<p>
  Давай подивимося, як це виглядає на практиці. Початковий код перетворюється на набір простих покрокових інструкцій, які легко читає інтерпретатор:
</p>

<code class="code">
  // Наш початковий JavaScript-код
  function add(a, b) {
    return a + b;
  }

  /* Приблизний вигляд згенерованого байт-коду в V8 (Ignition).
    Інтерпретатор виконує ці інструкції послідовно:
  */
  LdaSmi a   // Завантажити значення змінної 'a' в регістр-акумулятор
  Star r0    // Зберегти поточне значення акумулятора у тимчасовий регістр 'r0'
  LdaSmi b   // Завантажити значення змінної 'b' в регістр-акумулятор
  Add r0     // Додати поточне значення акумулятора (b) до того, що лежить у 'r0' (a)
  Return     // Повернути підсумковий результат з акумулятора
</code>

<h3>3. Оптимізація (JIT-компіляція)</h3>

<p>
  Виконуючи байт-код, рушій паралельно аналізує його виконання та збирає статистику. На основі цих даних він може прийняти рішення про те, що певні ділянки коду варто оптимізувати, скомпілювавши їх у машинний код.
</p>

<p>
  <strong>Профайлер (Profiler / Monitor)</strong> - один з компонентів рушія. Він рахує частоту викликів функцій. Функції, які виконуються багаторазово (наприклад, всередині циклів), позначаються як "гарячі" (hot).
</p>

<p>
  Сам інтерпретатор за допомогою механізму <strong>Inline Caches (IC)</strong> відстежує, які типи даних передаються у функції. Наприклад, якщо функція завжди отримує числа, це важлива інформація для оптимізації.
</p>

<p>
  Після цього, ґрунтуючись на зібраній інформації, рушій робить припущення — це називається <strong>спекулятивна оптимізація</strong>.
</p>

<p>
  Якщо <strong>Profiler</strong> бачив, що функція завжди отримує на вхід лише числа, <strong>TurboFan</strong> скомпілює машинний код, заточений виключно під роботу з числами, відкинувши перевірки типів.
</p>

<p>
  Оптимізований код замінює інтерпретований байт-код, і подальші виклики функції виконуються вже з високою продуктивністю.
</p>

<p>
  Однак, якщо в майбутньому функція почне отримувати інші типи даних (наприклад, рядки), оптимізований код може стати неефективним. У цьому випадку рушій може "деоптимізувати" функцію, повернувшись до інтерпретованого байт-коду. Це називається <strong>Деоптимізація (Bailout / Deoptimization)</strong>.
  <br>
  Деоптимізація — це дуже ресурсомістка операція. Тому рушії намагаються уникати її, використовуючи різні евристики для визначення, коли оптимізувати код.
</p>

<p class="info info--blue">
  Важливо розуміти, що оптимізація — це безперервний процес. Рушій постійно збирає дані і може повторно оптимізувати код, якщо ситуація зміниться.
</p>

<h3>Як писати код, який рушій зможе оптимізувати</h3>

<p>
  Як розробник, ти можеш допомогти JIT-компілятору не виконувати зайвої роботи і не йти в деоптимізацію.
</p>

<p>
  <strong>Дотримуйся мономорфізму:</strong> Намагайся передавати в конкретну функцію аргументи одного й того самого типу.
</p>

<p>
  <strong>Зафіксуй форму об'єктів (Приховані класи):</strong> Рушії створюють під капотом "приховані класи" (Hidden Classes / Shapes) для об'єктів з однаковим набором ключів. Завжди ініціалізуй властивості об'єкта в конструкторі і строго в одному й тому самому порядку.
</p>

<p>
  <strong>Уникай оператора <code>delete</code>:</strong> Видалення властивостей з об'єкта "ламає" його прихований клас, змушуючи рушій створювати новий і відкидати попередні оптимізації доступу до пам'яті.
</p>

<p>
  <strong>Не змінюй типи елементів масивів:</strong> Якщо масив почав свій життєвий цикл як масив чисел (наприклад, <code>[1, 2, 3]</code>), не пуш у нього рядки або об'єкти. Це змусить рушій перебудувати масив у пам'яті за менш ефективним шаблоном.
</p>
