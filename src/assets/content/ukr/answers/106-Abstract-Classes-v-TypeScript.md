<h3>Що таке абстрактний клас (Abstract Class)?</h3>
<p><span class="accent">Абстрактний клас</span> — це базовый клас, який не можна інстанціювати безпосередньо (створити через <code>new</code>). Він слугує «кресленням» для інших класів.</p>

<p>Його головна фішка в тому, що він може містити як оголошення методів (абстрактні методи, які зобов'язані реалізувати спадкоємці), так і готову реалізацію (звичайні методи, які спадкоємці отримують з коробки).</p>

<p class="info info--orange">При проектуванні архітектури вибір між абстрактним класом та інтерфейсом — це класична дилема. У TypeScript обидва інструменти надають потужні можливості для визначення контрактів і забезпечення типової безпеки, але вони слугують різним цілям і мають свої унікальні особливості. У цій відповіді ми розглянемо ключові особливості абстрактних класів у TypeScript.</p>

<h3>Коли використовувати Абстрактні класи замість Інтерфейсів?</h3>
<h4>Використовуй абстрактні класи, коли:</h4>

<p>1. <strong>Є спільна базова логіка.</strong> У тебе є кілька споріднених класів, які виконують частину роботи однаково. Цю спільну роботу (реалізацію) ти виносиш в абстрактний клас, щоб уникнути дублювання коду (принцип DRY).</p>

<p>2. <strong>Потрібен стан.</strong> Базовій сутності потрібно зберігати якісь внутрішні змінні або прапорці, які будуть використовуватися у всіх спадкоємцях.</p>

<p>3. <strong>Потрібні модифікатори доступу.</strong> Тобі потрібно приховати частину внутрішньої логіки базового класу від зовнішнього світу, використовуючи protected, залишивши її доступною лише для класів-спадкоємців.</p>

<p>4. <strong>Dependency Injection (Особливо актуально для фреймворків).</strong> Оскільки інтерфейси зникають у JavaScript-рантаймі, їх не можна використовувати як токени для впровадження залежностей. Абстрактні класи залишаються в JS, тому чудово працюють як DI-токени (частий трюк у серйозній архітектурі).</p>

<h4>Використовуй інтерфейси, коли:</h4>

<p>1. Потрібно просто описати «форму» об'єкта (наприклад, типізувати JSON-відповідь від API).</p>

<p>2. Потрібно задати суворий контракт поведінки для класів, які взагалі ніяк не пов'язані один з одним логічно (наприклад, інтерфейс Logger, який може бути реалізований і сервісом, і компонентом).</p>

<p>3. Потрібно реалізувати множинне успадкування поведінки.</p>

<h3>Приклад із життя: Патерн "Шаблонний метод"</h3>
<p>Абстрактні класи ідеально підходять для патерна <strong>Template Method</strong>, де базовий алгоритм заданий, а деталі делеговані спадкоємцям.</p>

<code class="code">
  abstract class DataFetcher {
    // Спільний стан
    protected isFetching = false;

    // Готова реалізація (спільна для всіх)
    public async fetchAndProcess(): Promise<void> {
      this.isFetching = true;
      try {
        const data = await this.getData(); // Виклик абстрактного методу
        this.processData(data);            // Виклик абстрактного методу
      } finally {
        this.isFetching = false;
      }
    }

    // Спадкоємці ЗОБОВ'ЯЗАНІ реалізувати ці деталі
    protected abstract getData(): Promise<any>;
    protected abstract processData(data: any): void;
  }

  // Конкретна реалізація
  class UserFetcher extends DataFetcher {
    protected async getData(): Promise<any> {
      return fetch('/api/users').then(res => res.json());
    }

    protected processData(data: any): void {
      console.log('Обробка користувачів:', data);
    }
  }

  const fetcher = new UserFetcher();
  fetcher.fetchAndProcess(); // Використовуємо спільний алгоритм з унікальними деталями
</code>

<h3>Комбінування наслідування та контрактів</h3>

<p>
  У TypeScript ти можеш легко комбінувати наслідування та реалізацію контрактів. Головне правило, про яке потрібно пам'ятати: наслідувати (<code>extends</code>) можна лише один базовий клас, а ось реалізовувати (<code>implements</code>) — скільки завгодно інтерфейсів.
  <br>
  Це дає тобі гнучкість у проектуванні архітектури, дозволяючи використовувати абстрактні класи для загальної логіки та інтерфейси для опису форм і контрактів.
</p>

<code class="code">
  // Абстрактний клас з базовою логікою
  abstract class BaseService {
    protected log(message: string): void {
      console.log(`[BaseService] ${message}`);
    }
  }

  // Інтерфейс для контракту
  interface Notifiable {
    notify(message: string): void;
  }

  // Конкретна реалізація, яка наслідує від абстрактного класу та реалізує інтерфейс
  class EmailService extends BaseService implements Notifiable {
    notify(message: string): void {
      this.log(`Відправка email з повідомленням: ${message}`);
      // Логіка відправки email...
    }
  }

  const emailService = new EmailService();
  emailService.notify('Привіт, світ!');
</code>

<h3>У чому сила такого підходу?</h3>
<p>Ти отримуєш найкраще з двох світів. З одного боку, повторно використовуєш вже написаний код з абстрактного класу, а з іншого — дотримуєшся суворих контрактів, визначених інтерфейсами.</p>
