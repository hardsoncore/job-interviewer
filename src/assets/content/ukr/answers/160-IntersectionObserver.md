<p>
  <span class="accent">Intersection Observer API</span> — це сучасний веб-API, який дозволяє відстежувати перетин елемента з іншим елементом або з областю видимості (viewport). Він надає спосіб асинхронного спостереження за змінами в перетині елемента з його батьківським елементом або з областю видимості, що особливо корисно для оптимізації продуктивності та покращення користувацького досвіду.
</p>

<p class="info info--blue">
  Статус «кросбраузерного стандарту» API отримав у березні 2019 року. Сьогодні це абсолютно базовий функціонал, який сміливо можна використовувати в будь-яких сучасних браузерах.
</p>

<h3>Основні юзкейси Intersection Observer API:</h3>

<ul>
  <li>
    <strong>Ліниве завантаження (Lazy Loading):</strong> Зображень, iframe'ів або важких компонентів, коли вони потрапляють у зону видимості.
  </li>
  <li>
    <strong>Нескінченний скрол (Infinite Scrolling):</strong> Підвантаження нових даних при досягненні кінця списку.
  </li>
  <li>
    <strong>Анімації при скролі:</strong> Запуск CSS-анімацій рівно в той момент, коли елемент з'являється на екрані.
  </li>
  <li>
    <strong>Аналітика видимості:</strong> Відстеження того, чи побачив користувач рекламний банер або певний блок тексту.
  </li>
</ul>

<h3>Архітектура та основні поняття</h3>
<p>
  Щоб почати роботу, потрібно створити екземпляр <code>IntersectionObserver</code>, передавши в нього callback-функцію та об'єкт з налаштуваннями (options).
</p>

<h4>1. Налаштування (Options)</h4>

<ul>
  <li>
    <strong>root:</strong> Елемент, який виступає в ролі області перегляду (viewport). За замовчуванням — вікно браузера (<code>null</code>). Якщо вказуєш конкретний елемент, він має бути предком відстежуваного елемента.
  </li>
  <li>
    <strong>rootMargin:</strong> Відступи навколо root елемента. Працює як CSS margin (наприклад, <code>"10px 20px 30px 40px"</code>, <code>"50px"</code> або <code>"10%"</code>). Дозволяє розширити або звузити область спрацьовування. Корисно, якщо потрібно почати завантаження картинки за <code>50px</code> до того, як вона фактично з'явиться на екрані.
  </li>
  <li>
    <strong>threshold:</strong> Чутливість до перетину. Число або масив чисел від <code>0.0</code> до <code>1.0</code>. Вказує, який відсоток елемента має бути видимим, щоб спрацював callback.
    <br>
    Наприклад, <code>[0, 0.5, 1]</code> дасть можливість викликати callback при перетині елемента на 0%, 50% і 100%.
  </li>
</ul>

<h4>2. Callback-функція</h4>
<p>
  Функція, яка викликається при перетині. Приймає два аргументи:
</p>

<ul>
  <li><strong>entries:</strong> Масив об'єктів <code>IntersectionObserverEntry</code> (дані про кожен відстежуваний елемент).</li>
  <li><strong>observer:</strong> Посилання на сам екземпляр спостерігача.</li>
</ul>

<p>
  Головні властивості об'єкта <code>entry</code> (з масиву <code>entries</code>):
</p>

<ul>
  <li><strong>isIntersecting (boolean):</strong> чи перетинається target елемент з <code>root</code> прямо зараз з урахуванням <code>threshold</code>.</li>
  <li><strong>intersectionRatio (number):</strong> поточний відсоток видимості (від 0.0 до 1.0).</li>
  <li><strong>target (Element):</strong> сам DOM-елемент, за яким ведеться спостереження.</li>
  <li><strong>boundingClientRect (DOMRectReadOnly):</strong> розміри та позиція target.</li>
  <li><strong>intersectionRect (DOMRectReadOnly):</strong> розміри та позиція видимої частини target.</li>
  <li><strong>rootBounds (DOMRectReadOnly):</strong> розміри та позиція root.</li>
  <li><strong>time (DOMHighResTimeStamp):</strong> час, коли відбувся перетин (корисно для аналітики).</li>
</ul>

<h3>Приклад використання</h3>

<code class="code">
  // 1. Задаємо налаштування
  const options = {
    root: null, // viewport браузера
    rootMargin: '0px',
    threshold: 0.5 // Спрацює, коли видно 50% елемента
  };

  // 2. Створюємо callback
  const callback = (entries, observer) => {
    entries.forEach(entry => {
      if (entry.isIntersecting) {
        console.log('Елемент у зоні видимості!', entry.target);

        // Виконуємо логіку (наприклад, підвантажуємо картинку)
        // entry.target.src = entry.target.dataset.src;

        // ДУЖЕ ВАЖЛИВО: припиняємо спостереження, якщо дію потрібно виконати один раз
        // Це запобігає витокам пам'яті в SPA (Vue/Angular)
        observer.unobserve(entry.target);
      }
    });
  };

  // 3. Ініціалізуємо Observer
  const observer = new IntersectionObserver(callback, options);

  // 4. Знаходимо елемент і починаємо спостереження
  const targetElement = document.querySelector('.my-element');
  observer.observe(targetElement);

  // При знищенні компонента (наприклад, у ngOnDestroy або beforeDestroy)
  // не забувай викликати observer.disconnect() для повного очищення.
</code>

<h3>Механіка роботи Intersection Observer API</h3>

<h4>Коли викликається колбек?</h4>
<p>
  При створенні <code>IntersectionObserver</code>, ми будемо щоразу викликати колбек, який передали при створенні, коли виконується умова, описана в options (при перетині порогу <code>threshold</code>).
</p>

<p>
  Колбек отримуватиме масив об'єктів <code>IntersectionObserverEntry</code>, кожен з яких містить інформацію про те, чи перетинається елемент з <code>root</code>, наскільки він видимий (<code>intersectionRatio</code>) і сам елемент (<code>target</code>).
</p>

<p>
  Всередині колбека ми можемо перевірити, чи перетинається елемент за допомогою властивості <code>isIntersecting</code>. Якщо воно <code>true</code>, значить елемент видимий у зоні видимості (або всередині <code>root</code> елемента) на певний відсоток, який ми вказали в <code>threshold</code>.
</p>

<p>
   Працює в обидва боки — і коли елемент з'являється, і коли зникає із зони видимості.
</p>

<p>
  Працює нескінченно. Якщо елемент з'являється і зникає, колбек буде викликатися щоразу, коли виконуються умови перетину. Тому важливо контролювати, коли потрібно припинити спостереження (наприклад, після першого завантаження картинки), щоб не створювати зайве навантаження на браузер.
</p>

<h4>Масив entries. Що всередині?</h4>

<p>
  Це список лише тих елементів, за якими ми явно наказали стежити (через <code>observer.observe()</code>), і стан яких щойно змінився (вони перетнули <code>threshold</code>).
</p>

<p>
  Браузер збирає їх у масив для оптимізації: якщо під час швидкого скролу поріг перетнули відразу 10 елементів, браузер віддасть їх пачкою за один виклик колбека, щоб не блокувати потік.
</p>

<p>
  Всередині масиву лежать не чисті HTML-теги, а об'єкти-знімки стану. Сам DOM-вузол зберігається у властивості <code>entry.target</code>, а статус видимості — в <code>entry.isIntersecting</code>.
</p>

<h3>Best Practices</h3>

<ul>
  <li>Завжди відключай спостереження (<code>unobserve</code> або <code>disconnect</code>) для елементів, які більше не потрібні, щоб уникнути витоків пам'яті.</li>
  <li>Будь обережний з великою кількістю відстежуваних елементів — це може вплинути на продуктивність. Не створюй новий <code>IntersectionObserver()</code> для кожної картки у списку з 1000 елементів. Створи один інстанс і виклич <code>observer.observe(target)</code> у циклі для всіх карток.</li>
  <li>Хоч обчислення перетинів і йде асинхронно, сам callback виконується в головному потоці. Не роби всередині нього важких синхронних обчислень. Якщо потрібна складна логіка, використовуй requestIdleCallback або setTimeout.</li>
</ul>
