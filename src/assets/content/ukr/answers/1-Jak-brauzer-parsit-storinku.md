<p>Процес перетворення HTML, CSS та JS коду на інтерактивну сторінку на екрані називається <strong>критичним шляхом рендерингу (Critical Rendering Path)</strong>. Він складається з наступних послідовних кроків:</p>

<h3>1. Парсинг HTML та побудова DOM-дерева</h3>
<p>Щойно браузер отримує перші байти HTML-документа по мережі, запускається HTML-парсер.</p>

<p>Парсер читає код зверху вниз, перетворюючи теги на токени (відкриття тегу, контент, закриття тегу), а токени — на вузли (nodes).</p>

<p>З цих вузлів вибудовується <span class="accent">DOM (Document Object Model)</span> — деревоподібна структура, що описує вміст сторінки.</p>

<p class="info">Важливо: Парсинг HTML відбувається інкрементально (в міру завантаження), але може бути заблокований синхронними скриптами (<code>&lt;script&gt;</code>).</p>

<h3>2. Завантаження зовнішніх ресурсів та виконання JavaScript</h3>
<p>Зустрічаючи посилання на зовнішні ресурси (шрифти, картинки, CSS, JS), браузер відправляє запити на їх завантаження.</p>

<p>Особливу роль відіграє JavaScript:</p>

<p> - За замовчуванням <code>&lt;script&gt;</code> блокує парсинг HTML. Браузер зупиняє побудову DOM, завантажує скрипт, виконує його, і тільки потім продовжує парсити HTML.</p>

<p> - <code>defer</code> — скрипт завантажується у фоні, не блокуючи парсинг, і виконується строго після того, як весь HTML проаналізовано, зберігаючи порядок підключення скриптів.</p>

<p> - <code>async</code> — скрипт завантажується у фоні і виконується відразу ж після завантаження, призупиняючи парсинг HTML на час виконання. Порядок виконання не гарантується.</p>

<p>Сучасні браузери використовують Preload Scanner — фоновий процес, який "забігає" наперед парсера HTML, знаходить зовнішні ресурси і починає завантажувати їх заздалегідь.</p>

<h3>3. Парсинг CSS та створення CSSOM</h3>
<p>Щойно завантажується CSS-файл, браузер перетворює його на <span class="accent">CSSOM (CSS Object Model)</span> — деревоподібну структуру, що описує стилі для вузлів DOM.</p>

<p>На відміну від HTML, парсинг CSS не може бути інкрементальним, оскільки стилі, оголошені в кінці файлу, можуть перевизначати стилі з початку.</p>

<p>CSS блокує рендеринг (Render-Blocking). Браузер не почне відмальовувати сторінку, поки не буде побудований повний CSSOM, щоб уникнути "миготіння нестилізованого контенту" (FOUC).</p>

<h3>4. Побудова дерева рендерингу (Render Tree)</h3>
<p>Браузер об'єднує DOM та CSSOM у єдине дерево рендерингу (Render Tree).</p>

<p>Render Tree містить лише ті вузли, які будуть візуально відображені на екрані.</p>

<p>Теги на кшталт <code>&lt;head&gt;</code>, <code>&lt;meta&gt;</code>, <code>&lt;script&gt;</code> не потрапляють до цього дерева.</p>

<p>Вузли з властивістю <code>display: none</code> виключаються з дерева рендерингу.</p>

<p>Вузли зі стилями <code>visibility: hidden</code> або <code>opacity: 0</code> залишаються в дереві, оскільки вони продовжують займати місце на екрані (впливають на геометрію).</p>

<h3>5. Обчислення геометрії (Layout / Reflow)</h3>
<p>Маючи Render Tree, браузер знає, що малювати, але не знає, де.</p>
<p>На етапі <span class="accent">Layout (Компонування)</span> браузер проходить по дереву рендерингу зверху вниз і обчислює точні розміри (ширину, висоту) та координати кожного елемента на екрані (у пікселях) відносно в'юпорта.</p>

<p><span class="accent">Reflow (Перекомпонування)</span> — це процес, при якому браузер наново обчислює розміри та точне розташування елементів на сторінці. По суті, це повторне виконання кроку «Розрахунок макета» (Layout).</p>

<h3>6. Відмальовування (Paint)</h3>
<p>Браузер знає геометрію всіх елементів. На етапі <span class="accent">Paint</span> він починає заповнювати пікселі на екрані: малює текст, кольори, межі, тіні та фонові зображення. Відмальовування зазвичай відбувається на кількох незалежних шарах (layers).</p>

<h3>7. Композитинг (Composite)</h3>
<p>Оскільки елементи могли бути відмальовані на різних шарах (наприклад, елементи з <code>transform</code>, <code>opacity</code> або <code>z-index</code>), на останньому етапі браузер накладає ці шари один на один у правильному порядку на графічному процесорі (GPU), формуючи підсумкове зображення, яке бачить користувач.</p>

<h3>Ключові події завантаження</h3>
<p><code>DOMContentLoaded</code> — спрацьовує, коли вихідний HTML повністю завантажений та розпарсений у DOM, без очікування завантаження таблиць стилів, зображень та фреймів.</p>

<p><code>load</code> — спрацьовує, коли сторінка завантажена повністю, включаючи всі залежні ресурси (картинки, стилі тощо).</p>
