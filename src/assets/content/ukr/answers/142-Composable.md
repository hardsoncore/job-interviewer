<h3>Що таке Composable?</h3>

<p>
  <span class="accent">Composable</span> - це функція, яка використовує Composition API (<code>ref</code>, <code>reactive</code>, <code>computed</code>, <code>watch</code>, хуки життєвого циклу тощо) для інкапсуляції та повторного використання реактивного стану і логіки в компонентах Vue.js.
  <br>
  <strong>Composable</strong> дозволяє розробникам створювати абстрактні функції, які можуть бути легко інтегровані в різні компоненти, забезпечуючи чистіший і модульніший код.
</p>

<p>
  <strong>Composable</strong> - це не вбудована концепція Vue, а скоріше патерн проєктування / угода, яку розробники можуть використовувати для організації та повторного використання логіки у своїх застосунках.
</p>

<p class="info info--orange">
  Існує сувора угода спільноти щодо неймінгу <strong>composable</strong> - використання use-неймінгу (наприклад, <code>useFetch</code>, <code>useAuth</code>). Ця угода допомагає відразу ідентифікувати функції, які є composables, і відрізняти їх від звичайних утиліт або сервісів.
</p>

<h3>У чому відмінність Composable від звичайної утиліти (utils)?</h3>

<p>
  Важливо розуміти різницю між функціями без стану (stateless) і функціями зі станом (stateful).
</p>

<p>
  <strong>Звичайна утиліта (utility)</strong> — це чиста функція, яка приймає аргументи, виконує логіку (наприклад, форматування дати або складні математичні обчислення) і повертає результат. Вона нічого не знає про Vue і не зберігає в собі даних, що змінюються.
</p>

<p>
  <strong>Composable</strong> — це функція, яка використовує можливості Vue (Composition API) для управління реактивним станом або побічними ефектами (підписки на події, таймери через хуки життєвого циклу). Вона зберігає в собі логіку для перевикористання, яка може включати реактивні дані та взаємодіяти з життєвим циклом компонента.
</p>

<p class="info info--blue">
  По суті, головна відмінність полягає в наявності реактивності: як тільки звичайна функція-утиліта починає інкапсулювати всередині себе стан (додали <code>ref</code> або <code>reactive</code>) або використовувати хуки (<code>onMounted</code>), вона перетворюється на <strong>composable</strong>.
</p>

<h5>Приклад НЕ composable:</h5>

<p>
  За офіційною документацією Vue, Composable — це функція, яка інкапсулює та перевикористовує stateful logic (логіку зі станом).
  <br>
  Цей приклад не є класичним композаблом з трьох причин:
  <br>
  - Вона не створює і не інкапсулює в собі незалежний стан (немає своїх <code>ref</code> або <code>reactive</code>).
  <br>
  - Вона не повертає жодних реактивних даних для використання у компоненті.
  <br>
  - Порушено базову угоду про іменування - функція не починається з префікса use (має бути щось на зразок useLogger).
</p>

<code class="code">
  function logCount(count) {
    watch(count, (val) => console.log(val));
  }
</code>

<h5>Приклад composable:</h5>

<p>
  Ця функція використовує <code>ref</code> для створення реактивного стану і повертає його, що дозволяє компоненту, який викликає цю функцію, реагувати на зміни цього стану. Це робить її composable.
</p>

<code class="code">
  function useCounter() {
    const count = ref(0);

    const increment = () => count.value++;

    return { count, increment };
  }

</code>

<h3>Золоті правила написання Composables</h3>

<p>
  <strong>Неймінг:</strong> Ім'я функції завжди має починатися з use (наприклад, <code>useFetch</code>, <code>useMouse</code>, <code>useAuth</code>).
</p>

<p><strong>Інкапсуляція:</strong> Уся логіка, пов'язана з конкретним завданням, має бути всередині однієї функції.</p>

<p>
  <strong>Значення, що повертається:</strong> Повертай простий JavaScript-об'єкт, що містить <code>ref</code> (а не <code>reactive</code> об'єкт). Це дозволяє безпечно деструктурувати результат у компоненті, не втрачаючи реактивності.
</p>

<p>
  <strong>Робота з реактивними аргументами:</strong> Хороший композабл повинен вміти працювати з реактивними аргументами. Для "розпакування" переданих аргументів усередині функції варто використовувати <code>unref()</code> (або <code>toValue()</code> у Vue 3.3+). Це дозволяє композаблу реагувати на зміни вхідних даних.
</p>

<p>
  <strong>Прибирання за собою:</strong> Якщо композабл створює слухачі подій (event listeners) або таймери, він зобов'язаний очистити їх у хуку <code>onUnmounted</code>.
</p>

<h3>Базовий шаблон (на прикладі відстеження миші)</h3>

<code class="code">
  import { ref, onMounted, onUnmounted } from 'vue';

  export function useMouse() {
    // 1. Ініціалізація реактивного стану
    const x = ref(0);
    const y = ref(0);

    // 2. Логіка оновлення стану
    function update(event) {
      x.value = event.pageX;
      y.value = event.pageY;
    }

    // 3. Підписка на події при монтуванні
    onMounted(() => window.addEventListener('mousemove', update));

    // 4. Очищення при розмонтуванні (КРИТИЧНО ВАЖЛИВО)
    onUnmounted(() => window.removeEventListener('mousemove', update));

    // 5. Повернення ref-ів
    return { x, y };
  }
</code>

<h3>Локальний vs Глобальний стан</h3>

<p>
  <strong>Локальний (за замовчуванням):</strong> Стан (<code>ref</code>) оголошується всередині функції. Кожен компонент, що викликає useSomething(), отримає свою незалежну копію стану.
</p>

<p>
  <strong>Глобальний (Shared):</strong> Стан оголошується зовні функції, у замиканні модуля. Усі компоненти, що викликають композабл, ділитимуть один і той самий реактивний стан (міні-заміна Vuex/Pinia).
</p>

<code class="code">
  import { ref } from 'vue'

  // Глобальний стан (один на весь застосунок)
  const globalCount = ref(0)

  export function useCounter() {
    // Локальний стан (новий для кожного компонента)
    const localCount = ref(0)

    return { globalCount, localCount }
  }
</code>

<h3>Композиція всередині композиції</h3>

<p>
  Композабли можуть викликати інші композабли. Наприклад, useFetch може використовувати всередині себе useEventListener для відстеження статусу мережі (online/offline) перед запитом.
</p>

<h3>Коли використовувати Composables?</h3>
<p>
  <strong>Логіка, що повторюється:</strong> Коли у тебе є логіка, яка повторюється в декількох компонентах (наприклад, робота з формами, автентифікація, взаємодія з API), композабли дозволяють інкапсулювати цю логіку і легко повторно використовувати її.
</p>

<h3>Які переваги дає використання Composables?</h3>
<p>
  <strong>Повторне використання:</strong> Композабли дозволяють легко повторно використовувати логіку в різних компонентах, що зменшує дублювання коду.
</p>

<p>
  <strong>Чистота компонентів:</strong> Виносячи складну логіку в композабли, компоненти стають чистішими і легшими для читання, оскільки вони фокусуються тільки на відображенні.
</p>

<p>
  <strong>Гнучкість:</strong> Композабли можуть бути легко комбіновані один з одним, що дозволяє створювати складні функції з простіших.
</p>

<h3>Коли НЕ варто використовувати Composables?</h3>
<p>
  <strong>Проста логіка:</strong> Якщо логіка дуже проста і використовується тільки в одному місці, створення композабла може бути надлишковим.
</p>

<p>
  <strong>Вузькоспеціалізовані функції:</strong> Якщо функція призначена для дуже специфічного завдання, яке не має сенсу використовувати повторно, вона може не потребувати статусу композабла.
</p>

<h3>Ключові відмінності Composable від mixins та HOCs</h3>
<p>
  <strong>Mixins:</strong> Mixins можуть створювати проблеми з конфліктами імен та неявним походженням методів або залежностей (Implicit dependencies), тоді як composables повертають явні об'єкти і не мають таких проблем.
</p>
<p>
  <strong>HOCs (Higher-Order Components):</strong> HOCs обгортають компоненти і можуть створювати складні ієрархії, тоді як composables просто повертають об'єкти зі станом і логікою, які можна використовувати всередині компонентів без обгортання.
</p>
