<p class="info info--blue">
  Це оператори-перетворювачі, які роблять дві речі:
  <br>
  <br>
  <strong>Проекція (Mapping):</strong> На КОЖНЕ значення з «зовнішнього» потоку вони створюють новий «внутрішній» потік (inner Observable).
  <br>
  <strong>Управління підпискою (Flattening):</strong> Вони автоматично підписуються на ці внутрішні потоки і виводять їхні значення в загальний потік.
  <br>
  <br>
  Різниця між ними полягає лише в стратегії конкурентності: що робити, якщо прийшло нове значення, поки старий внутрішній потік ще не завершився?
</p>

<h3>mergeMap</h3>
<p>
  <strong>mergeMap</strong> (або flatMap) підписується на кожен внутрішній Observable та об'єднує їхні результати в один потік. Це означає, що всі внутрішні Observable виконуються паралельно, і їхні результати змішуються в одному потоці.
</p>

<p>
  <strong>Коли використовувати:</strong> незалежні операції, порядок не важливий — завантаження файлів, збереження непов'язаних записів, websocket-фрейми.
</p>
<p>
  <strong>Потенційні проблеми:</strong> при високочастотному outer-потоці можна створити нескінченну кількість відкритих HTTP-запитів. Використовуй <code>mergeMap(fn, concurrency)</code>, щоб обмежити паралелізм.
</p>

<code class="code">
  import { fromEvent, interval } from 'rxjs';
  import { mergeMap, take } from 'rxjs/operators';

  const clicks = fromEvent(document, 'click');
  const result = clicks.pipe(
    mergeMap((ev) => interval(1000).pipe(take(3)))
  );
  // При кожному кліку запускається новий interval, який
  // генерує послідовність чисел (0, 1, 2 тощо) щосекунди
  // всі виконуються паралельно

  result.subscribe(x => console.log(x));
  // Вивід: при кожному кліку виводитиметься 0, 1, 2 з інтервалом в 1 секунду,
  // і ці потоки будуть змішуватися
</code>

<h3>concatMap</h3>
<p>
  <strong>concatMap</strong> підписується на кожен внутрішній Observable, але виконує їх послідовно. Він чекає, поки поточний внутрішній Observable завершиться, перш ніж підписатися на наступний. Це гарантує порядок виконання, але може бути повільнішим, якщо внутрішні Observable виконуються довго.
</p>

<p>
  <strong>Коли використовувати:</strong> анімації (наступна — після закінчення поточної), послідовні операції із залежностями, міграція даних, відтворення дій.
</p>
<p>
  <strong>Потенційні проблеми:</strong> якщо inner-Observable ніколи не завершується (<code>complete</code> не приходить), черга заблокована назавжди.
</p>

<code class="code">
  import { fromEvent, interval } from 'rxjs';
  import { concatMap, take } from 'rxjs/operators';

  const clicks = fromEvent(document, 'click');
  const result = clicks.pipe(
    concatMap((ev) => interval(1000).pipe(take(3)))
  );
  // При кожному кліку запускається новий interval, який
  // генерує послідовність чисел (0, 1, 2 тощо) щосекунди

  // Але на відміну від mergeMap, ці потоки виконуються послідовно:
  // наступний починається тільки після завершення попереднього

  result.subscribe(x => console.log(x));
  // Вивід: при кожному кліку виводитиметься 0, 1, 2 з інтервалом
  // в 1 секунду, але наступний набір чисел почнеться тільки після завершення попереднього
</code>

<h3>switchMap</h3>
<p>
  <strong>switchMap</strong> підписується на кожен внутрішній Observable, але скасовує попередній, якщо новий Observable приходить до завершення попереднього. Це корисно для сценаріїв, де важливий лише останній результат, наприклад, при пошукових запитах.
</p>

<p>
  <strong>Коли використовувати:</strong> пошукові запити за вводом, автодоповнення, навігація за роутами — будь-який сценарій, де старий результат втрачає актуальність. Саме тому це оператор за замовчуванням у більшості Angular HttpClient-патернів.
</p>
<p>
  <strong>Потенційні проблеми:</strong> не використовуй з мутуючими операціями (POST, DELETE). Якщо запит на сервер був скасований у RxJS, мережевий запит браузера все одно полетів — скасування відбувається лише на рівні підписки на відповідь.
</p>

<code class="code">
  import { fromEvent, interval } from 'rxjs';
  import { switchMap, take } from 'rxjs/operators';

  const clicks = fromEvent(document, 'click');
  const result = clicks.pipe(
    switchMap((ev) => interval(1000).pipe(take(3)))
  );
  // При кожному кліку запускається новий interval, який
  // генерує послідовність чисел (0, 1, 2 тощо) щосекунди

  // Але якщо клік відбувається до завершення поточного interval,
  // попередній потік буде скасовано і почнеться новий

  result.subscribe(x => console.log(x));
  // Вивід: при кожному кліку виводитиметься 0, 1, 2 з інтервалом в
  // 1 секунду, але якщо клікнути знову до завершення поточного потоку,
  // він буде скасований і почнеться новий
</code>


<h3>exhaustMap</h3>
<p>
  <strong>exhaustMap</strong> підписується на перший внутрішній Observable та ігнорує всі наступні, поки поточний не завершиться. Це корисно для запобігання одночасному виконанню кількох внутрішніх Observable, наприклад, при обробці кліків.
  <br>
  Поки активний поточний inner-Observable, всі нові значення з outer просто ігноруються.
</p>

<p>
  <strong>Коли використовувати:</strong> submit-кнопки (double-submit protection), логін, оплата — коли повторний клік до завершення першого має бути проігнорований.
</p>
<p>
  <strong>Потенційні проблеми:</strong> через те що натискання втрачаються, користувач може не отримати зворотного зв'язку. Краще комбінувати з блокуванням UI на час запиту.
</p>

<code class="code">
  import { fromEvent, interval } from 'rxjs';
  import { exhaustMap, take } from 'rxjs/operators';

  const clicks = fromEvent(document, 'click');
  const result = clicks.pipe(
    exhaustMap((ev) => interval(1000).pipe(take(3)))
  );
  // При кожному кліку запускається новий interval, який
  // генерує послідовність чисел (0, 1, 2 тощо) щосекунди

  // Але якщо клік відбувається до завершення поточного interval,
  // новий потік буде проігноровано

  result.subscribe(x => console.log(x));
  // Вивід: при кожному кліку виводитиметься 0, 1, 2 з інтервалом в
  // 1 секунду, але повторні кліки до завершення поточного потоку будуть ігноруватися
</code>
