<h3>Function Declaration (Оголошення функції)</h3>

<p>Це класичний спосіб створення функції. Вона оголошується окремою конструкцією в основному потоці коду.</p>

<code class="code">
  function greet(name) {
    return `Hello, ${name}!`;
  }
</code>

<h4>Ключові особливості:</h4>

<p><strong>Ім'я обов'язкове:</strong> Функція повинна мати ім'я (greet), щоб парсер JS зрозумів, що це саме Declaration.</p>

<p><strong>Hoisting (Підняття):</strong> Це найважливіша механіка. Рушій JavaScript сканує код до його виконання і «піднімає» всі Function Declarations на самий верх їхньої області видимості (глобальної або функціональної).</p>

<p><strong>Використання:</strong> Ідеально підходить для створення глобальних утиліт або основних логічних блоків, які мають бути доступні в будь-якому місці файлу.</p>

<code class="code">
  // Функція викликається ДО її оголошення в коді — і це працює.
  const total = getDiscount(100);

  function getDiscount(price) {
    return price * 0.9;
  }
</code>

<p class="info info--orange">У стрілочних функцій немає можливості бути Function Declarations — вони завжди є Function Expressions.</p>

<h3>Function Expression (Функціональний вираз)</h3>

<p>
  Це спосіб створення функції, при якому функція присвоюється змінній. Вона може бути анонімною (без імені) або мати ім'я (Named Function Expression).
  <br>
  JS сприймає таку функцію просто як значення.
</p>

<code class="code">
  // Анонімний функціональний вираз
  const greet = function(name) {
    return `Hello, ${name}!`;
  };
</code>

<h4>Ключові особливості:</h4>

<p><strong>Ім'я необов'язкове:</strong> Зазвичай вони анонімні (як у прикладі вище), але можуть бути й іменованими (Named Function Expression).</p>

<p><strong>Відсутність Hoisting:</strong> Функція створюється рівно в той момент, коли потік виконання (execution flow) доходить до цього рядка. Її не можна викликати до ініціалізації.</p>

<p><strong>Використання:</strong> Замикання, колбеки, передача функцій як аргументів, методи об'єктів.</p>

<h3>Приклад помилки (TDZ — Temporal Dead Zone):</h3>

<code class="code">
  // ReferenceError: Cannot access 'getDiscount' before initialization
  const total = getDiscount(100);

  const getDiscount = function(price) {
    return price * 0.9;
  };
</code>

<h3>Named Function Expression (NFE)</h3>

<p>Різновид Function Expression, де у функції є локальне ім'я. Воно доступне лише всередині самої цієї функції. Це критично корисно для рекурсії та зрозумілих стек-трейсів при дебагінгу.</p>

<code class="code">
  const factorial = function calc(n) {
    if (n <= 1) return 1;
    return n * calc(n - 1); // Звернення до самої себе за внутрішнім ім'ям
  };

  console.log(factorial(5)); // 120
  console.log(calc(5)); // ReferenceError: calc is not defined
</code>


<h3>Важливі нюанси та рекомендації</h3>

<p><strong>Консистентність:</strong> Намагайся не змішувати стилі без необхідності. Для допоміжних функцій у корені файлу використовуй Declaration (щоб вони не захаращували верхню частину файлу, їх можна прибрати вниз). Для логіки, що залежить від умов, або колбеків — Expression.</p>

<p><strong>Стрілочні функції (Arrow Functions):</strong> Пам'ятай, що в сучасному фронтенді багато Function Expressions замінюються на стрілочні функції (<code>const sum = (a, b) => a + b</code>). Вони теж є виразами і не спливають (no hoisting), але додатково не мають власного контексту this (що часто рятує при передачі колбеків у компонентах).</p>

<p><strong>Суворий режим (<code>'use strict'</code>):</strong> У суворому режимі Function Declaration має блочну область видимості. Якщо оголосити її всередині <code>if {}</code>, зовні цього блоку вона буде недоступна.</p>

<p><strong>Рекурсія:</strong> Якщо функція повинна викликати саму себе, і ти використовуєш Function Expression, краще зробити її Named Function Expression, щоб уникнути проблем з доступом до функції всередині неї самої.</p>

<p><strong>Продуктивність:</strong> Здебільшого різниці в продуктивності між цими способами немає. Вибирай стиль, який краще підходить для конкретного завдання і покращує читабельність коду.</p>

<p class="info info--blue">
  Хоча існує міф про те, що велика кількість Function Declarations може уповільнювати виконання коду, на практиці сучасні рушії JavaScript оптимізують їх виконання, і різниця в продуктивності мінімальна.
  <br>
  Різниця в часі ініціалізації між тисячею Function Declarations і тисячею Function Expressions мізерна і не піддається адекватному вимірюванню в реальних умовах.
</p>

<h3>Під капотом: Рушій та Специфікація ECMAScript</h3>

<p>Згідно зі специфікацією ECMAScript, коли JavaScript-рушій (наприклад, V8 у Chrome) виконує код, він проходить дві фази.</p>

<h4>Фаза 1: Creation Phase (Фаза створення / компіляції)</h4>

<p>До того як буде виконано хоча б один рядок коду, рушій парсить скрипт (переводить його в токени), будує AST (абстрактне синтаксичне дерево) і виділяє пам'ять під змінні та функції. Це називається ініціалізацією Lexical Environment (лексичного оточення).</p>

<p>Під час цієї фази всі Function Declarations обробляються і їхні імена додаються в область видимості. Це означає, що навіть якщо функція оголошена внизу файлу, вона буде доступна вгорі.</p>

<p>Function Expressions, з іншого боку, не обробляються на цьому етапі. Рушій бачить лише оголошення змінної (наприклад, <code>const sum = ...</code> або <code>var sum = ...</code>).</p>

<h3>Різниця на рівні граматики мови (AST)</h3>
<p>У специфікації ECMAScript це дві принципово різні синтаксичні конструкції:</p>

<p><strong>Statement (Інструкція / Твердження):</strong> Function Declaration — це statement. Вона виконує дію (реєструє функцію в оточенні), але не повертає значення. Її не можна передати як аргумент або присвоїти змінній безпосередньо.</p>

<p><strong>Expression (Вираз):</strong> Function Expression — це частина виразу. Вираз у JS — це будь-який шматок коду, який повертає значення. Рушій обчислює FE і повертає значення — саму функцію (об'єкт у пам'яті), яку ми потім можемо присвоїти змінній, передати в setTimeout або використовувати в замиканні.</p>

<p>Класичний сеньйорський приклад на розуміння граматики:</p>

<code class="code">
  // Це Function Declaration. Викличе синтаксичну помилку (SyntaxError),
  // оскільки statement не може бути викликаний на місці, а парсер очікує ім'я функції.
  function() {
    console.log('test');
  }();
  // SyntaxError: Function statements require a function name

  // Обертаємо в дужки (Grouping Operator).
  // Дужки в JS можуть містити тільки вирази (Expressions).
  // Парсер миттєво перетворює FD на Function Expression.
  // Тепер це IIFE (Immediately Invoked Function Expression), і код працює.
  (function() {
    console.log('test');
  })();
</code>

<h3>Міф: Function Declaration гальмує ініціалізацію (Performance)</h3>
<p>Суть міфу: Оскільки всі Function Declarations спливають (Hoisting) та ініціалізуються в пам'яті на фазі створення контексту (до старту виконання коду), величезна їх кількість нібито сильно сповільнить старт застосунку.</p>

<h4>Як працює насправді (Lazy Parsing):</h4>
<p>Сучасні JS-рушії (V8, SpiderMonkey) неймовірно оптимізовані. Щоб не витрачати час і пам'ять на старті, вони використовують механізм ледачого парсингу (Lazy / Pre-parsing).</p>

<p><strong>Pre-parsing на старті:</strong> Коли рушій сканує код і знаходить Function Declaration, він не компілює її тіло цілком. Він робить лише швидкий прохід: перевіряє базовий синтаксис і реєструє ім'я функції в лексичному оточенні. Це займає частки мілісекунд.</p>

<p><strong>Lazy Compilation:</strong> Повноцінний парсинг тіла функції, побудова повного AST і генерація байт-коду відбуваються тільки в момент першого виклику цієї функції. Якщо функція ніколи не буде викликана, рушій так і не витратить ресурси на її глибоку компіляцію.</p>
